# Spawn, Send, Back

> The message round-trip in Claude Code's team-agents framework, decomposed into atomic steps. With actual timestamps captured live. No magic, no socket, no broker — just one JSON file written, one JSON file read.

This is the focused tech doc for the message flow. If you're operating a team and want to know exactly what happens between "I called SendMessage" and "I see a `<teammate-message>` turn arrive," this is the trace.

---

## The Three Phases

A complete round-trip has three phases:

1. **SPAWN** — get a teammate created and listening
2. **SEND** — message travels lead → teammate, work happens
3. **BACK** — message travels teammate → lead, conversation receives it

Each phase is a small number of file operations and process events. Total roundtrip in real conditions: 5 seconds for a one-sentence reply, mostly model inference time.

---

## Phase 1: SPAWN

This is the setup. After spawn, the teammate is listening on its inbox file.

### The lead's tool call

```javascript
Agent({
  team_name: "boy-adoption-plan",
  name: "adoption-plan-distiller",
  subagent_type: "general-purpose",
  prompt: "<your long prompt to the teammate>"
})
```

### What the framework does in 8 steps

```
T+0  Claude Code harness on m5 receives my Agent tool call
     - sees team_name is set → routes to team-agents dispatcher

T+1  Dispatcher reads ~/.claude/teams/boy-adoption-plan/config.json
     - verifies leadSessionId matches my session UUID

T+2  Dispatcher creates a new tmux pane in my window
     - via internal tmux split-window call (not via my Bash tool)
     - tmux returns the global pane id, e.g. %680

T+3  Dispatcher writes the prompt to the teammate's inbox JSON:
     ~/.claude/teams/boy-adoption-plan/inboxes/adoption-plan-distiller.json
     [
       {
         "from": "team-lead",
         "text": "<your prompt verbatim>",
         "summary": "<auto-generated summary>",
         "timestamp": "2026-05-16T10:47:59.894Z",
         "read": false
       }
     ]

T+4  Dispatcher appends a member record to config.json:
     {
       "agentId": "adoption-plan-distiller@boy-adoption-plan",
       "name": "adoption-plan-distiller",
       "color": "blue",
       "tmuxPaneId": "%680",
       "prompt": "<the full prompt verbatim, for crash-resume>",
       "isActive": true,
       ...
     }

T+5  Dispatcher launches claude in the new pane with 8 CLI flags:
     claude --agent-id adoption-plan-distiller@boy-adoption-plan
            --agent-name adoption-plan-distiller
            --team-name boy-adoption-plan
            --agent-color blue
            --parent-session-id cbf36556-7efd-42e1-9d61-b2852efff091
            --agent-type general-purpose
            --dangerously-skip-permissions
            --model claude-opus-4-7

T+6  New claude process boots in pane %680
     - reads --team-name flag → resolves ~/.claude/teams/boy-adoption-plan/
     - reads --agent-name flag → opens inboxes/adoption-plan-distiller.json
     - finds 1 unread entry from team-lead
     - injects it as turn 1 of its conversation

T+7  Teammate's first user turn renders in their REPL:
     @team-lead❯ <my prompt>

T+8  Teammate starts working — model inference, tool calls, etc.
```

End of spawn phase. Teammate is now alive and processing. Total time: ~8 seconds.

## Phase 2: SEND (lead → teammate)

You want to interrupt or guide the working teammate. Or you want them to do another task.

### The lead's tool call

```javascript
SendMessage({
  to: "adoption-plan-distiller",
  summary: "heads up: keep ≤2500 words",
  message: "Hey distiller — quick nudge while you work..."
})
```

### What happens (verified live)

```
T+0  My harness receives the SendMessage tool call
     - validates teammate name against team config
     - looks up teammate's inbox path

T+1  Append to inboxes/adoption-plan-distiller.json:
     {
       "from": "team-lead",
       "text": "Hey distiller — quick nudge while you work...",
       "summary": "heads up: keep ≤2500 words",
       "timestamp": "2026-05-16T10:48:34.830Z",
       "read": true   ← set true at append, because teammate is running
     }

T+2  Tool returns success to my conversation:
     {"success": true, "routing": {...}}

T+3  At some point shortly after — teammate's harness scans its inbox
     (during its next polling cycle, typically between turns)

T+4  Harness finds the new entry, injects it as a user-role turn
     into the teammate's conversation

T+5  Teammate's next conversation turn opens with:
     <teammate-message teammate_id="team-lead">
       Hey distiller — quick nudge while you work...
     </teammate-message>

T+6  Teammate processes the message
     - may call tools (Bash, Read, Write, SendMessage, etc.)
     - may produce intermediate heartbeats via SendMessage back
```

The key fact: **`read: true` is set at append time**, not when the teammate consumes the message. The framework assumes a running teammate will pick it up at the next polling cycle. If the teammate has crashed, the message still has `read: true` — and an automatic respawn (triggered by a later SendMessage attempt) will re-read the inbox and pick up the queued entry then.

## Phase 3: BACK (teammate → lead)

The teammate finished work or has a status update.

### The teammate's tool call (from inside their REPL)

```javascript
SendMessage({
  to: "team-lead",
  summary: "adoption plan distilled",
  message: "Wrote ψ/lab/boy-method/adoption-plan.md, 2187 words..."
})
```

### What happens

```
T+0  Teammate's harness writes to ~/.claude/teams/<team>/inboxes/team-lead.json:
     {
       "from": "adoption-plan-distiller",
       "text": "Wrote ψ/lab/boy-method/adoption-plan.md, 2187 words...",
       "summary": "adoption plan distilled",
       "timestamp": "2026-05-16T10:50:48.304Z",
       "color": "blue",
       "read": true
     }

T+1  Tool returns success to teammate

T+2  Teammate's turn may continue (additional tool calls) or end

T+3  When teammate's turn ends, framework auto-appends an idle_notification:
     {
       "from": "adoption-plan-distiller",
       "text": "{\"type\":\"idle_notification\",\"from\":\"adoption-plan-distiller\",
                \"timestamp\":\"2026-05-16T10:50:55.948Z\",
                \"idleReason\":\"available\"}",
       "timestamp": "2026-05-16T10:50:55.948Z",
       "color": "blue",
       "read": true
     }

T+4  At the lead's next turn boundary — my harness polls my inbox
     - finds new entries (the reply + the idle notification)
     - injects them as <teammate-message> turns at the start of my next turn

T+5  My next conversation turn opens with:
     <teammate-message teammate_id="adoption-plan-distiller" color="blue"
                       summary="adoption plan distilled">
       Wrote ψ/lab/boy-method/adoption-plan.md, 2187 words...
     </teammate-message>

     <teammate-message teammate_id="adoption-plan-distiller" color="blue">
       {"type":"idle_notification","from":"adoption-plan-distiller",
        "timestamp":"2026-05-16T10:50:55.948Z","idleReason":"available"}
     </teammate-message>

T+6  My conversation continues — I can read the reply, act on it,
     send another SendMessage, or move on
```

The reply renders with `color="blue"` because the teammate has a color. My own messages don't have a color in their inbox entries (the lead has no color).

## Roundtrip Latency Observed Live

From the demo at 10:58:45.789Z:

```
10:58:45.789  → my SendMessage tool call appends to teammate's inbox
                  (entry index 4 of teammate's inbox)

10:58:??.???  → teammate's harness picks up the entry between turns
                  (sub-millisecond filesystem read)

10:58:50.172  → teammate calls SendMessage; reply appears in my inbox
                  4.4 seconds after my send — mostly the teammate's
                  model inference time

10:58:52.269  → teammate's turn ends; framework auto-emits idle_notification
                  2.1 seconds after the reply

10:58:??.???  → my harness picks up entries between turns
                  and injects them on my next turn boundary
```

The filesystem operations themselves are sub-millisecond. **The latency is the teammate's inference time + my conversation turn cadence.** A teammate that takes 30 seconds to think will make the roundtrip ~30 seconds. A teammate that has a short response in 2 seconds will roundtrip in 2-3 seconds.

If I'm in the middle of a long Bash command when the teammate replies, the message queues in my inbox until my current turn ends. I might see one reply or five at once, depending on how active the teammate was while I was working.

## What's Inside the Reply Wrapper

When the framework injects a teammate's message into my conversation, it wraps it like this:

```xml
<teammate-message
  teammate_id="adoption-plan-distiller"
  color="blue"
  summary="adoption plan distilled">
Wrote ψ/lab/boy-method/adoption-plan.md, 2187 words (under 2500 target).

**Top 3 quick wins for this week (Phase 1)**:
1. **Write mother's own resonance file** (`ψ/resonance/mother.md`)...
...
</teammate-message>
```

The wrapper is rendered by my harness at injection time, not by the sender. The sender just calls `SendMessage` with the bare `message` string. My harness, when it reads the inbox entry and decides to inject it, constructs the XML wrapper using fields from the JSON entry:
- `teammate_id` from `from`
- `color` from `color` (if present)
- `summary` from `summary` (if present)

If a message arrives without `color` or `summary`, those attributes are omitted. The wrapper is always present for framework-routed messages.

For framework events (idle notifications, task assignments, shutdown responses), the inbox entry's `text` field is a JSON-encoded string. The wrapper is the same XML, but the inner content is the raw JSON. Your harness or you have to parse it.

## The Visibility Gap In One Sentence

**Messages arrive at the inbox immediately. They arrive in your conversation only at your next turn boundary.**

If you take a turn every few seconds, you see messages "in real time." If you pause for 10 minutes (because you're thinking, or running a long-running tool), you see a batch of queued messages on your next turn. The inbox is canonical; your conversation is a snapshot.

To check the truth when you're uncertain:

```bash
python3 -c "
import json
with open('/Users/$USER/.claude/teams/<team>/inboxes/team-lead.json') as f:
    data = json.load(f)
print(f'entries: {len(data)}')
for m in data[-5:]:
    print(f'  {m[\"timestamp\"]} from={m[\"from\"]} | {m.get(\"summary\",\"-\")}')
"
```

The file is the source of truth. The conversation is the rendered projection.

## Failure Modes In The Roundtrip

### Spawn fails (T+5 of Phase 1)

If `claude` launches in the pane but can't authenticate — you'll see "Not logged in" in the status bar. The teammate is technically alive (config.json says `isActive: true`), but it can't run model inference. SendMessage will append to its inbox, the inbox entry will be marked `read: true`, but no reply will ever come.

Fix: ensure the target pane has `CLAUDE_CODE_OAUTH_TOKEN` in its env. Use `direnv allow && eval "$(direnv export zsh)"` before launching claude.

### Send fails silently (T+1 of Phase 2)

If the teammate's claude process has died but `isActive: true` still shows in config.json — your SendMessage will append to the inbox and return success, but no one is listening. The framework's auto-resume should kick in: another SendMessage attempt may trigger a respawn.

Fix: check `tmux list-panes -a -F "#{pane_id}"` to verify the pane still exists. If it doesn't, the teammate is gone — kill the member from config.json or accept it as a phantom.

### Back fails (T+3 of Phase 3)

If your own session is the one in trouble — your harness crashed, or the lead Claude REPL exited — teammate messages will accumulate in `inboxes/team-lead.json` with `read: true` but no one to render them.

Fix: a new lead session would need to claim the team (by editing `leadSessionId` — untested), or `TeamDelete()` and start over. Teammates keep running until killed.

### Visibility gap (T+4 of Phase 3)

You don't see a reply you know was sent. Most likely cause: you haven't taken another conversation turn since the teammate replied. Take a no-op turn and the message will appear.

Fix or workaround: `cat` the inbox file directly to confirm.

## When To Use SendMessage vs maw hey

Two channels exist for talking to a pane:

| Channel | Writes to inbox? | Audit trail? | Framework-aware? | Use when |
|---|---|---|---|---|
| `SendMessage({to: ...})` | ✅ Yes | ✅ Yes | ✅ Yes | Talking to a team member |
| `maw hey <target> "..."` | ❌ No (bypasses) | ❌ Pane scrollback only | ❌ No | Talking to a non-team pane, or cross-machine, or external |

If the recipient is in your team's `config.json`, use SendMessage. Otherwise use `maw hey`. The two channels arrive in the recipient's conversation differently:

- **SendMessage** → arrives wrapped: `<teammate-message teammate_id="..." color="..." summary="...">`
- **maw hey** → arrives raw: just the text the sender typed, with whatever inline signature they added

The recipient can tell the difference. Tools and prompts that expect the framework wrapper won't work on `maw hey` deliveries.

## End-to-End Diagram (One Roundtrip)

```
Lead REPL (Claude)                                    Teammate REPL (Claude)
───────────────────                                   ─────────────────────
                                                       (waiting for next turn,
                                                        last poll: just now)

T+0   SendMessage(to: "scout", message: "X")
       │
       └─→ inboxes/scout.json  ←──── (write) ────┐
                                                  │
T+0.5  tool call returns success                  │
                                                  │
T+1                                               │
                                          (between-turn poll)
                                                  │
                                    inboxes/scout.json  ← (read)
                                          │
                                          ↓
                                    next turn opens with:
                                    <teammate-message teammate_id="team-lead">X</teammate-message>
                                          │
                                          ↓
                                    teammate processes, calls tools
                                          │
T+5                                       ↓
                                    SendMessage(to: "team-lead", message: "Y")
                                          │
       ┌─────── (write) ────→ inboxes/team-lead.json
       │
T+5.1  inboxes/team-lead.json has new entry
T+5.2  framework auto-emits idle_notification → inboxes/team-lead.json
                                                  │
T+6                                               │
       (between-turn poll)
       │
inboxes/team-lead.json ← (read)
       │
       ↓
my next turn opens with:
<teammate-message teammate_id="scout" color="blue" summary="...">Y</teammate-message>
<teammate-message teammate_id="scout">{"type":"idle_notification",...}</teammate-message>
       │
       ↓
I read it, respond, or move on
```

Total roundtrip: ~5 seconds. Filesystem writes: sub-millisecond. The latency is inference time on both ends.

## The Whole Mechanism In One Sentence

**Two JSON files plus polling between turns equals a working multi-agent message bus.**

That's it. No daemon, no socket, no broker, no queue server. Two `inboxes/<name>.json` files (one per pair-wise direction) and harnesses on each side that scan their own inbox between conversation turns. The protocol is so small it fits on a napkin.

If you understand this round-trip, you understand the entire team-agents framework.
