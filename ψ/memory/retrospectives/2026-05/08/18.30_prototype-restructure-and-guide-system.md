## Honest Feedback

**Friction 1: Over-engineering the guide.** I built 5 iterations of PrototypeGuide in one session, each adding complexity that created new bugs. The trigger was Un's reference to the prototype-d2 control panel — I tried to replicate its full step-navigation + callout system in React with portal-based beacons and DOM observation. The d2 reference was a static HTML prototype with simple scene switching; my implementation tried to do the same thing reactively against a live app with modals, sheets, and dynamic content. Fundamentally different problem. I should have recognized this gap before the first iteration, not after the fifth.

**Friction 2: Not testing in browser myself.** Un had to tell me repeatedly that it was broken. I relied on HTTP status codes (200 = "it works") without actually opening the browser and clicking through the flow. When I finally ran Playwright automated tests, they revealed real issues (modal overlay blocking control bar, element selectors not found). I should have done this before reporting completion.

**Friction 3: Session length vs. focus.** 7 hours is too long for one session without a context checkpoint. By hour 5, I was debugging guide sync issues while the actual goal (UX work on lab order flow) hadn't started. The restructure + index page took ~3 hours and was high-value. The guide took ~3 hours and ended up stripped to basics. Should have timeboxed the guide to 1 hour and moved to UX.

## Lessons Learned

1. **Start minimal, add complexity only when the simple version proves insufficient.** The guide's final form (80 lines, beacons + bar) could have been v1. Everything between was waste.
2. **PM owns execution.** Don't say "send to Chrome." Say "I'll spawn a frontend agent." Specialists advise; PM's team-agents execute.
3. **Nested folder convention solves the sync problem.** `{base-ui}/{activity}` means base changes propagate to all activities. Convention + cross-domain imports + shared _components = no duplication.
