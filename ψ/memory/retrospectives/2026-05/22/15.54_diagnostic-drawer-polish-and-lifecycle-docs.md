## AI Diary

This session taught me something about the difference between documenting what exists and designing what should exist. We started with a simple merge of two timeline steps, but the moment the user asked me to create lifecycle docs for all entities, the act of writing them down exposed structural problems I hadn't noticed.

The biggest reveal was when the user asked "Queue status เท่ากับ OPD status ไหม?" during the /grill-me. I'd been treating WAITING_LAB as a natural Queue status, but when I tried to apply the same Queue pipeline to Grooming, it broke down. WAITING_LAB, WAITING_DISPENSING, CANCELLED_TREATMENT are all OPD-specific concerns leaking into a generic pipeline. The user saw this before I did.

The wait_for_result discussion was equally revealing. Two identical actions ("สั่งตรวจเลือด") with completely different lifecycle implications depending on intent. The user's instinct to use SOAP section as an implicit flag was elegant, and then immediately caught the edge case with slash commands being universal. That kind of UX thinking, where the system adapts to where you are rather than asking you to declare intent, is what separates good clinical software from form-filling.

I also learned about SVG diagram overlap the hard way. Hand-crafted SVG coordinates are brittle. Every fix created new overlaps. The step-by-step approach (10 small diagrams instead of 2 massive swim lanes) was the right call, but the text positioning still needed multiple iterations. White knockout rects helped but weren't a silver bullet.

## Honest Feedback

**Friction 1: SVG overlap iterations.** I spent too many rounds fixing text overlaps in the overall-flow.html. The approach of having agents write SVGs with absolute coordinates is inherently fragile. Each fix moved one label but created a new overlap. A better approach might be to use a layout engine (even just CSS grid with positioned divs) or a proper diagramming library that handles text placement automatically. The manual SVG coordinate dance cost probably 4-5 round trips.

**Friction 2: Explore agents hitting context limits.** 3 of 4 research agents failed with "Prompt is too long" when exploring the codebase. The prototype files are large (queue/_mock.ts, SOAPContent.tsx, design-system/page.tsx) and reading them fills context fast. I should have used more targeted grep commands from the start instead of delegating broad exploration to subagents.

**Friction 3: As-Is vs Target ambiguity.** The overall-flow.html tries to show both current implementation and proposed architecture, but the proposed changes (Queue generic pipeline, OPD own status, PAUSED, wait_for_result, target_opd_id) haven't been validated with the backend team. We're documenting aspirational architecture in the same file as current state, which could confuse readers who don't know which is real.

## Lessons Learned

1. **Documentation as architecture tool** - Writing lifecycle docs forces you to confront structural inconsistencies. The Queue/OPD status separation only became visible when we tried to document the flow for non-OPD services.

2. **Implicit UX > explicit configuration** - The SOAP section as wait_for_result flag is better than a toggle because it matches clinical workflow intent. Design for the context the user is already in.

3. **Small diagrams > one big diagram** - Step-by-step focused SVGs are more readable than comprehensive swim lanes, even though they show less at once. Readers can hold one relationship in mind at a time.
