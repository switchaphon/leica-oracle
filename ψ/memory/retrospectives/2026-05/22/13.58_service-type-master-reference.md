## AI Diary

Today was one of those sessions where the grill format proved its worth beyond doubt. What started as "update service terminology" could have gone sideways fast — there were two Figma files with conflicting taxonomies, three different color schemes in the codebase, and two naming conventions (lowercase vs SCREAMING_SNAKE) fighting for dominance. The grill forced us to walk every branch of the decision tree before touching a single line of code.

The breakthrough came at Question 2, when Un started explaining the IPD lifecycle — how it always originates from OPD or Surgery, how it's an admission state not a service category. That insight cascaded: if IPD isn't a service type, then neither is Lab (it's an order), and Boarding/Hotel are a booking flow. Suddenly the 14-item mess collapsed into a clean 3×N model. The Figma chip design's 5 colors became 3 — not because we compromised, but because the extra colors were solving the wrong problem.

What humbles me is the bug Un caught immediately — the chips showed "รักษาทั่วไป" instead of the subtype name. I'd built the component correctly (it accepts subtypeKey), wired the appointment page correctly, but didn't add service_subtype to the queue mock data. The root cause was obvious in hindsight: I was so focused on the type-level refactor that I forgot the original design agreement. Un saw it in one screenshot. That's why the human reviews.

The OPD refactor at the end was a bonus — Un asked a simple question ("why don't these share base UI?") and we eliminated 425 lines of duplication in 15 minutes. The old call-to-opd/opd/[id] page had an entire inline MockSOAPContent that was a stale copy. Now both routes use the same OpdPageBody and SOAPContent, differing only in data source and navigation.

## Honest Feedback

1. **Figma Desktop Bridge friction** — The Bridge plugin needs to be manually opened in each Figma file, and there's no way to check from the CLI which files have it running. I had to ask Un to open the plugin twice, which broke flow. A "try to connect, fall back to REST API screenshot" pattern would save round-trips.

2. **Subagent work sometimes misses intent** — The parallel agents did a good job on the mechanical changes (replacing NEUTRAL_PILL, updating imports) but missed the semantic issue (subtypeKey not being passed). The agents followed the literal brief but didn't have the grill context about "chip text = subtype name." I should have included that requirement explicitly in every agent brief rather than assuming they'd infer it from the component API.

3. **Mock data as a second-class citizen** — Every prototype refactor requires touching mock data in multiple files. The queue mock, dashboard mock, and appointment mock each have their own ServiceType definition and mock items. There's no single mock seed that generates consistent data across pages. This is fine for now but will bite harder as more pages join.

## Lessons Learned

- **Grill before implement, always** — The 13-question grill saved us from building a 5-color system that would need immediate refactoring. 30 minutes of questions prevented hours of rework.
- **Service taxonomy: type ≠ lifecycle** — Service types answer "what service?", lifecycle states answer "where is the case now?" IPD/Boarding are lifecycle, not service types. This distinction matters for data modeling.
- **Subtype labels on chips, not type labels** — When color already encodes the category, text should carry the specific information (subtype name). Redundant labeling wastes the most valuable UI real estate.
