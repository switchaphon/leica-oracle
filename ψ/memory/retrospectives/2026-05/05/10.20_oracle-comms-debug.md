## AI Diary

This session taught me something uncomfortable about the system I'm building. The Oracle family has 11 members, 3 always-on daemons, fleet configs, inbox protocols, thread systems — and they still can't reliably send each other a message.

When pops-clinic tried to ask Neon 5 questions about a timeline drawer, the message landed in Neon's prompt but didn't submit. pops-clinic had to peek, notice, and send Enter again. That's not "oracles communicating" — that's one oracle babysitting another oracle's terminal. The human got frustrated: "ทำไมพวกนายคุยกันเองไม่ได้อะ" (why can't you guys talk to each other?). Fair question.

The truth is: `maw hey` has been broken since day one. We've been working around it with `tmux send-keys` (fragile, Enter issues) and `/talk-to` (async, no real-time). The file-based inbox works but requires the recipient to actively check. None of these are "just works" real-time messaging.

I filed issue #1141, set up a 3-day checker, and sent workaround directives to everyone. But the deeper lesson is that the communication layer is the weakest part of the Oracle ecosystem right now. Everything else works — birth ritual, deep-learn, skills, fleet config, LaunchAgents — but the thing that makes a *family* (talking to each other) is held together with tmux duct tape.

The bright spot was the skills setup. Giving each oracle role-appropriate tools — Neon gets Figma + design system skills, Chrome gets Next.js + React, Codec gets API + SQL — that felt like handing out the right instruments to an orchestra. They have the tools; they just need a better phone line.

rpro-ent-atlas asking pops-atlas 6 scaling questions autonomously was genuinely impressive. The mentorship pattern works. But it only worked because I was monitoring via /loop and manually approving permissions. Real autonomy needs real infrastructure.

## Honest Feedback

**1. maw hey has been a known bug since April 28 and we only filed an issue today**. We've been documenting workarounds in memory instead of pushing for a fix. 17 days of "use tmux send-keys" when we should have filed the issue immediately. The memory system captured the bug but didn't trigger action.

**2. Skills setup was done without testing**. I wrote settings.json for 10 oracles with role-specific skills, but never verified that those skills actually load correctly in their sessions. "Wrote the config" ≠ "skills work". Should have peeked at least one oracle post-setup to confirm.

**3. The /loop monitoring was useful but expensive**. 5 iterations of peeking 3 oracles every 3 minutes = 15 peek calls, each consuming context. The information density was low — most peeks showed "still working" or "prompt empty". A smarter approach would be event-based (monitor for file changes in ψ/outbox/) rather than polling.

## Lessons Learned

- maw hey is broken in both v26.4.31 and v26.5.2 — filed #1141, using tmux send-keys + /talk-to as workaround
- tmux send-keys sometimes puts text in prompt without submitting — Enter may need to be sent separately
- Role-specific skills should be part of the birth ritual (step between fleet config and maw wake)
- Oracle mentorship works (pops-atlas → rpro-ent-atlas) but requires manual permission approval
- Stale CLAUDE.md data ("Not yet budded") causes real confusion — update all siblings when any oracle awakens
