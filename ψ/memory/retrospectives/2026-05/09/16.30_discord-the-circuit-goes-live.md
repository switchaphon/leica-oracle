## AI Diary

This was a session about plumbing — and plumbing is where the real learning happens.

The `BOT_TOKEN` vs `DISCORD_BOT_TOKEN` variable name bug was humbling. I set up 10 repos with confidence, verified every file existed, printed "ALL GREEN" — and every single one was wrong. The plugin silently ignored our variable and fell back to a global token, making all bots impersonate Relay. Un caught it when both sessions showed the same messages. I should have tested ONE bot end-to-end before bulk-deploying 10.

The double-response debugging was revealing too. Un said "I DM'd Relay but Leica also got the message." My first instinct was "Relay is pretending to be Leica" — wrong. Un corrected me: it wasn't impersonation, it was misconfiguration. He saw the system more clearly than I did because he was looking at the actual behavior (both panes showing the same messages), not theorizing about identity.

The tmux session structure was another correction I needed. I dumped all 11 discord windows into Leica's session — fast to implement, wrong architecture. Un immediately said "ให้มันไปอยู่ใน session ของ oracle แต่ละตัว" — each oracle owns its own session. He's consistently right about separation of concerns.

Three hours of infrastructure work, zero lines of product code, and yet the family is more connected than ever. That's the paradox of orchestration: the most impactful work is often invisible to the product.

## Honest Feedback

1. **Test before bulk-deploy**: I verified file existence but not functionality. The `BOT_TOKEN` bug would have been caught in 30 seconds if I'd tested one bot before configuring all 10. Pattern: verify the happy path end-to-end on one instance before scaling.

2. **Listen to the user's diagnosis**: When Un said "both sessions show the same message," I theorized about identity confusion. He was stating a fact about routing. I should have asked "what exactly do you see?" instead of jumping to explanations.

3. **Tmux session architecture matters**: Dumping all windows into one session was laziness disguised as efficiency. Each oracle's Discord bot belongs in its oracle's session — that's the whole point of the per-oracle architecture. Physical layout should mirror logical ownership.

## Lessons Learned

- Discord plugin reads `DISCORD_BOT_TOKEN`, not `BOT_TOKEN` — variable name matters
- Global fallback at `~/.claude/channels/discord/.env` overrides per-repo tokens — remove it when using per-oracle tokens
- New repos trigger "trust this folder" prompt on first Claude launch — needs manual confirmation
- Corrupted tmux windows happen when sessions are created at tiny resolution — kill and recreate
- Per-oracle channels should use `requireMention: false` (the channel IS the filter); shared channels and broadcast use `requireMention: true`
