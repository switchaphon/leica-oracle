## Key Code Changes

### 1. 70/30 Split with KPI Inside Left Column
Previous layout had KPI full-width above the split. Un's idea (option B): move KPI into the 70% column so calendar aligns at the same level on the right. Weekly calendar (1 row) instead of full month (6 rows) — saves ~200px vertical space.

### 2. Date-Reactive Cascade
Single `selectedDate` state → `dateAppts` memo → cascades to:
- `filteredAppointments` (table)
- `computeSummary()` (KPI cards)
- `next3DaysCount` (KPI card 3)
- `sidebarAppointments` (timeline, vet-filtered)
- `apptDaySet` (calendar dots)
- `dateLabel` (tab bar + title)

### 3. Gradient Heatmap Bars
```
HEAT_GRADIENT = 'linear-gradient(to right, #16A34A, #CA8A04, #F07A00, #B91C1C)'
backgroundSize: `${(maxScale / v.count) * 100}% 100%`
```
The `backgroundSize` trick stretches the full gradient across the theoretical max, so a 50% bar shows green→yellow and a 100% bar shows the full green→red sweep. All 4 colors from DS semantic palette.

### 4. Sidebar Height = Table Height
`items-stretch` on the flex parent makes both columns equal height. Timeline card uses `flex-1 min-h-0` + `absolute inset-0` scroll area to fill remaining space after calendar + workload. Gradient fade at bottom hints scrollability.

## Architecture Decisions

- **Weekly vs monthly calendar**: Un chose weekly — sufficient for appointment scheduling context, matches dashboard, saves space for workload + timeline
- **Optional date field**: `date?: string` on Appointment avoids touching 15 existing mock entries — fallback `a.date || TODAY` treats undated appointments as today
- **Heatmap max scale**: `Math.max(actual_max, 6)` — prevents 1-appointment days from showing red bars. 6 = realistic "half-day" baseline

## Honest Feedback

**1. DS-first discipline is still weak.** I reached for `hsl()` interpolation for the heatmap when the DS already has Success/Warning/Error tokens. Un had to correct me. This is the third or fourth time this pattern has appeared — I invent new colors instead of checking what exists. The fix isn't just "remember to check DS" — it's to make DS consultation the first step of any visual decision, before writing any style code.

**2. Hook ordering in large components is fragile.** The appointment page is now 870+ lines with ~15 hooks. Inserting a new `useMemo` in the wrong position caused a TDZ crash. In production this would be caught by tests, but prototypes have no tests (by convention). I need to be more deliberate about reading the full hook chain before inserting new ones.

**3. Screenshot resolution makes verification uncertain.** Playwright CLI screenshots at 1920×1200 are small enough that subtle issues (dot overlap, gradient colors, text truncation) are hard to verify. The user's actual browser screenshot (the one showing the sidebar overflow) revealed the problem clearly. I should ask the user to verify in-browser more often instead of relying solely on automated screenshots.

## Lessons Learned

1. **DS palette is the source of truth for all colors** — never introduce HSL/RGB without checking semantic tokens first
2. **`items-stretch` + `flex-1 min-h-0` + `absolute inset-0`** — the standard recipe for "sidebar card fills remaining height with scroll"
3. **`backgroundSize` gradient trick** — stretch a full-spectrum gradient across theoretical max, partial fill reveals proportional color slice
4. **Optional field + fallback > mass-editing** — `date?: string` with `a.date || TODAY` adds multi-day mock data without touching existing entries
5. **Hook insertion order matters** — in a 800+ line component, always trace the dependency chain before adding a new useMemo
