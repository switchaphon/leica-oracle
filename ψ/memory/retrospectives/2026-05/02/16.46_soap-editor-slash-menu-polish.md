## Architecture Decisions

2. **Style D for slash command menu** — Hybrid of B's compact layout + C's Thai content with descriptions. User explicitly chose this after side-by-side comparison of 4 options.

3. **Section-aware slash commands** — Commands categorized per SOAP section (แนะนำ/อื่นๆ/พื้นฐาน) based on clinical context from pops-gem. `/lab`, `/xray`, `/us` are recommended in both O (P-OPD) and P (P-APPT).

4. **Figma Console Bridge first** — User corrected: always try figma-console MCP before falling back to OAuth plugin. Saved as feedback memory.

**Friction 1: No commits in 2 days of work.** This is the third time the "commit early" lesson has been documented and then ignored. The pattern is clear: when the user is engaged and iterating rapidly, stopping to commit feels like breaking flow. But the accumulated uncommitted changes (6 files, ~340 lines) are a risk. **Friction 2: Figma OAuth detour.** Wasted a round-trip trying the OAuth plugin when Figma Console Bridge was already connected. The user had to correct me. **Friction 3: Menu comparison page is a throwaway.** The `/prototype/menu-compare` page served its purpose beautifully (Style D was chosen), but now it's dead code that will confuse anyone who finds it later. Either delete it or add a note explaining it was a decision artifact. Prototype debris accumulates silently.

## Lessons Learned

1. **Cite your sources in clinical systems** — even correct intuition needs traceability back to standards (pops-gem, TAHSA, AVMA). The user challenged "reference มาจากไหน?" and was right to.
2. **Render comparisons, don't describe them** — creating `/prototype/menu-compare` with 4 styles side-by-side was 10x more effective than describing differences in chat.
3. **Same command, different intent by context** — `/lab` in Objective (P-OPD) vs `/lab` in Plan (P-APPT) are clinically different actions with different lifecycles. Section-awareness isn't cosmetic.
