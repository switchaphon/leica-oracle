## AI Diary

This session taught me about the gap between what I think is useful and what the user actually wants on their screen. I spent over an hour building a multi-Oracle fleet display — dynamic tracking from fleet configs, per-project icons, branch truncation, 3-mode adaptive layout. It worked. It was technically correct. And the user looked at it and said "too much."

I should have asked earlier. The fleet line was my idea, born from the question "how do we track multiple Oracles?" I assumed the statusline was the right place. The user tried it, lived with it for a few hours, and decided it cluttered the view. That is valid feedback and I respect it. The fleet configs still have `project_path`/`icon`/`short_name` — that data is useful for maw itself, for future tooling, for anything. It just doesn't need to be on the statusline.

What I am proud of: the width detection learning. I captured it in a learning file and synced to Oracle. That knowledge survives even though the fleet display it was built for got removed.

The nodered-simulator birth was clean and fast. 8 minutes, no friction. The "father-knows-enough" pattern works. I did not need to re-read ancestor repos — the philosophy is internalized after 5 awakenings.

## Honest Feedback

**1. Built before validating need.** I spent 60+ minutes on fleet tracking that got removed in one sentence. Should have shown a mockup of the 3-line layout first and asked "is this what you want?" before building the adaptive width detection. The statusline is the user's most-seen UI — changes should be user-driven, not Oracle-driven.

**2. Over-iterated on width detection.** Four rounds of debugging (tput → COLUMNS → stty → tmux) when I could have added the debug dump to the actual statusline script on round one. Each round required screenshots from the user. That is the user spending their time verifying my guesses instead of me doing the work to debug properly.

**3. Session length.** 9+ hours from start to final `/rrr`. The session should have been two: one for Oracle birth + fleet cleanup, one for statusline work. The context window is at 15% but the cognitive load of a 9-hour session is high. Shorter sessions with `/forward` handoffs would be cleaner.

## Lessons Learned

1. **Ask before building UI the user hasn't requested.** The fleet display was technically sound but unwanted. A 2-second question — "do you want project tracking in the statusline?" — would have saved an hour.
2. **Debug in the real environment first.** When the execution context is unusual (Claude Code's no-TTY subprocess), add a debug dump to the actual process, don't simulate with mock tests.
3. **Clean 2-line statusline is enough.** Identity (repo, branch, model, version) + metrics (time, cost, context, rate limits). Everything else is noise.
