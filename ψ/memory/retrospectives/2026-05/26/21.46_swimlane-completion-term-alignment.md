## Honest Feedback

**Friction 1: SVG coordinate math is the bottleneck.** Every diagram requires extensive mental calculation for node positions, arrow endpoints, text baselines, and box widths for Thai text. The coordinate-comment-block technique helps, but there's no tooling to validate positions without rendering. I spent significant thinking time on viewBox dimensions and annotation box sizing for the diagnostic visit_no diagrams.

**Friction 2: The term rename (target_opd_id → target_visit_no) should have been a single atomic operation across ALL docs.** Instead it happened incrementally — first in diagnostic-flow SVGs, then caught during overall-flow review. The "keep existing content" instruction from Un created a dual-term situation in diagnostic-flow.html itself (SVGs use visit_no, prose uses opd_id). This will confuse future readers unless resolved.

**Friction 3: The Vercel plugin hook fires on every Read/Edit of files in the app/ directory, suggesting Next.js/Vercel skills that are completely irrelevant for static HTML flow docs.** This adds noise to every operation — "You must run Skill(nextjs)" when I'm editing a raw HTML file with zero Next.js involvement.

## Lessons Learned

1. **Cross-doc term alignment needs a sweep pass**: When a field name changes in one doc, immediately grep all sibling docs and update atomically.
2. **Concrete data formats (VN69-05-XXXX) make diagrams 10x more useful than abstract placeholders (A, B)**: The reader sees real system behavior, not theory.
3. **SQL inside SVG annotations works surprisingly well**: For data model flows, showing the exact query alongside the visual makes the mechanism unambiguous.
