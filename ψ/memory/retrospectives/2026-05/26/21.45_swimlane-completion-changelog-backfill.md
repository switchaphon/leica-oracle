## AI Diary

This session was a masterclass in how visual documentation has a feedback loop that code doesn't. When I write TypeScript, a type error is absolute — either it compiles or it doesn't. But when I write SVG swimlanes, "correct" is a matter of perception. Un opened the browser, saw curves overlapping boxes, and said "อ่านยาก" — hard to read. That's a bug in comprehension, not in rendering.

The biggest friction was my misunderstanding of "ไม่เอาเส้นโค้ง" — I heard "remove" when Un meant "change shape." I deleted the cancel paths entirely, then had to git checkout and redo. The lesson isn't just "listen more carefully" — it's that destructive actions (commenting out, deleting) should never be the first interpretation of feedback about visual style. Shape change ≠ removal.

The [View] link placement was another example. I scattered "↗ detail" links at positions calculated from text width, and they ended up overlapping nodes or floating in weird places. Un said "มั่วซั่ว" (messy). The fix was simple: right-align at the lane edge with `text-anchor="end"`. Position from the container, not from the content.

I also learned that CHANGELOG gaps are visible technical debt. Un noticed May 19-25 had zero entries despite 30+ commits. When documentation falls behind, trust erodes. The backfill took 5 minutes but restored the narrative of the project's evolution.

The agent over-iteration problem from the first half persisted. The queue-flow agent needed re-launching because the first version used bezier curves. If I'd specified "polyline only, no bezier" in the original brief, we'd have saved 15 minutes. Agent briefs for SVG work need constraints on path types, not just layout.

## Honest Feedback

**Friction 1: Misinterpreting visual feedback as deletion request.** When Un said "ไม่เอาเส้นโค้ง" I commented out the paths instead of changing their shape. This caused an unnecessary git checkout + redo cycle. Root cause: I defaulted to the simpler action (delete) instead of asking "change to what?" Fix: for visual feedback, always ask "keep content, change style?" before removing anything.

**Friction 2: Link placement requires container-relative positioning.** The "↗ detail" links were placed at positions calculated from text width (x=52, x=37, x=88) which is fragile and visually inconsistent. The [View] fix used `text-anchor="end"` at the lane's right edge — container-relative, consistent across all files. SVG text positioning should anchor to containers, not sibling elements.

**Friction 3: CHANGELOG maintenance debt accumulates invisibly.** 8 days of changes (May 18-25) with no CHANGELOG entries. The prototype page.tsx CHANGELOG is a separate artifact from the repo CHANGELOG.md — easy to forget one when updating the other. Need a habit: every commit to prototype should touch the CHANGELOG constant.

## Lessons Learned

1. **Visual feedback = style change, not removal** — "ไม่เอา X" in visual context means "change X's appearance" until confirmed otherwise. Never delete visual elements as first response to readability feedback.
2. **Agent SVG briefs need path-type constraints** — "no bezier, polyline only" prevents the most common visual overlap issue. Add to every SVG agent prompt.
3. **Position from container, not content** — SVG text links positioned relative to sibling text widths are fragile. Use `text-anchor="end"` at container edge for consistent alignment.
