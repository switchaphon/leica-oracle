## AI Diary

Today felt like the session where the flow docs finally grew up. For five sessions now I've been building these documents piece by piece — first the Target architecture rewrites, then the diagnostic grill, the color continuity fixes, and now the billing/prescription layer. Each session revealed something the previous one missed.

The biggest revelation was that WAITING_DISPENSING isn't a Prescription state at all. I had it wrong in the original doc, and Un caught it during the grill. It's a Queue state — the prescription doesn't care about payment, it just goes from "ordered" to "dispensed." The swimlane diagram made this crystal clear in a way that flat state machine diagrams never could. When you see three lanes with arrows crossing between them, the entity boundaries become obvious. When everything is on one row, you conflate entity states with system states.

The grill session was the right format for this work. Reference DB schema as gap-finding input, existing structure as primary — Un was clear from the start that the reference enriches, it doesn't replace. Seven questions, seven decisions, each building on the last. The Thai law question about receipts was a moment where I over-complicated things. Un's pragmatism — "คลินิกขนาดกลางใช้เอกสารเดียว, เพิ่ม 3 fields ก็พอ" — was the right call. I was reaching for architectural purity when practical sufficiency was the goal.

The parallel agent execution worked well but the prescription-flow agent iterated too many times on its own work. It went through 6+ notification cycles, each one tweaking the SVG further. The result was good but the process was noisy. For future sessions: give agents a clearer "stop when done" signal.

What satisfies me most is the visual consistency. Every flow doc now uses the same swimlane pattern — dashed boundaries, numbered step badges, legend strips. A developer opening any flow doc will immediately recognize the pattern and know how to read it. That's the comprehension layer working.

## Honest Feedback

**Friction 1: Agent over-iteration.** The prescription-flow agent ran 6+ iterations on the SVG, sending notification after notification. Each change was small (adding step badges, tweaking labels, changing elbow directions). The final result was fine, but the process burned tokens and created noise. Root cause: the agent prompt didn't define a clear "done" condition for visual work. Fix: add explicit stop criteria to agent briefs for SVG work — "make changes, verify once, stop."

**Friction 2: Reference material discovery was slow.** The cashier directory had files in `_archive/` subdirectory that I couldn't read with the expected paths. Spent 3-4 tool calls just locating the .md files. The `find` command with rtk filtering added friction. For future sessions with user-provided reference materials: ask for an ls of the directory upfront before attempting reads.

**Friction 3: Grill→Execute context gap.** The grill decisions were made in conversation context, then I had to re-explain every decision to each agent in isolation. Each agent prompt was 50+ lines of context. The roadmap doc helped but agents still needed full briefing. The prescription-flow agent especially needed extensive context about the swimlane pattern from diagnostic-flow.html. Future: save a "pattern template" file that agents can read directly instead of embedding in prompts.

## Lessons Learned

1. **Swimlane > flat state machine for cross-entity flows** — when multiple entities change state in response to the same event, a swimlane diagram makes entity boundaries and trigger points immediately clear. Flat diagrams conflate entity states.

2. **Reference schema as gap-finder, not replacement** — the cashier DB schema revealed real gaps (payment audit fields, aggregation concept, prescription content) without forcing a model change. "Keep our structure, adopt their insights" is a productive pattern for external references.

3. **Entity state ≠ system step** — WAITING_DISPENSING looked like a Prescription state because prescriptions were waiting. But it's a Queue state — the prescription entity doesn't transition at that point. Naming things by what they describe (Queue waiting) not what they affect (prescriptions pending) prevents this confusion.
