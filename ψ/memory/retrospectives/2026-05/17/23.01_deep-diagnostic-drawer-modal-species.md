## Key Code Changes

### 1. Timeline Injection Pattern
```tsx
timeline[0].info = (<><SourceTag .../>{testSummary}</>);
timeline[0].action = { label: 'ดูรายการตรวจ' };
timeline[0].onAction = () => setOrderModalOpen(true);
```
Post-generation mutation of timeline entries to inject rich UI without changing the data builder.

### 2. Modal Size Constants
```ts
export const MODAL_SIZE = { S: 'max-w-sm', M: 'max-w-lg', L: 'max-w-2xl', XL: 'max-w-5xl' } as const;
export const MODAL_WIDTH = { S: 384, M: 512, L: 672, XL: 1024 } as const;
```
Dual export: Tailwind classes for DialogContent, numeric widths for Modals component.

### 3. Species Display Convention
```tsx
// Before: <SpeciesIcon species={pet.species} />
// After:  pet.species === 'DOG' ? 'สุนัข' : 'แมว' (in metadata string)
```
Icons stay in tables for quick scan; Thai text in headers/drawers/modals for information density.

### 4. Inline X Button
```tsx
<DialogContent hideCloseButton>
  <div className='flex items-center gap-2'>
    <DialogTitle>...</DialogTitle>
    <div className='flex-1' />
    <button onClick={onClose}><X className='h-4 w-4' /></button>
  </div>
```
Moves close button from absolute positioning into flex row for natural alignment with title and chips.

## Architecture Decisions

1. **Timeline as primary navigation** — The drawer is now timeline-first. All detail sections (test list, clinical notes, provider info) live inside timeline steps or open as modals from them. This follows the queue drawer pattern already proven in the codebase.

2. **Modal as work-order document** — The OrderDetailModal is designed to be self-contained: a lab tech or receptionist can read it without needing context from the drawer. This means some info is "duplicated" (pet info appears in both drawer and modal) — intentional for standalone readability.

3. **Variant-driven density** — PatientHeader uses `variant` prop to control density, not design language. Full = avatar + owner row. Compact = no avatar, owner on own row. Minimal = single inline text. Same visual identity across all.

4. **Token before pattern** — Modal sizes were tokenized before being used in a second place. This prevents the ad-hoc `max-w-*` drift that already existed in the codebase.

## Honest Feedback

**Friction 1: Screenshot tax.** Every visual verification required writing a CJS script, running it, reading the image. When it failed (3 times this session), the retry cycle was painful. A pre-built `screenshot(url, selector?)` helper in the project would save 2-3 minutes per verification. Over a 4-hour session with 15+ screenshots, that's 30+ minutes of infrastructure overhead.

**Friction 2: Scattered pet rendering.** The species icon removal touched 8 files because pet identity is rendered inline in each page rather than through a single shared component. PatientHeader covers some cases, but queue/call-to-opd and pet/page have their own card layouts with inline pet rendering. A `<PetIdentityLine pet={pet} />` component that outputs "HN... ชื่อ · สุนัข · breed · gender · age · weight" would have reduced this to a 1-file change.

**Friction 3: Modal header alignment.** The shadcn Dialog close button uses `absolute right-4 top-4`, which doesn't naturally align with flex row content. Every custom modal header requires either `pr-10` padding hack or `hideCloseButton` + manual X button. This should be a project convention documented in the DS — "always use hideCloseButton + inline X in header flex row."

## Lessons Learned

1. **Timeline card collapse** — Proven pattern: collapse >200px detail sections into timeline step info cards with action buttons opening full-detail modals
2. **Modal size tokens early** — Define size constants before the second modal uses them; ad-hoc max-w creates silent inconsistency
3. **Species: text > icon** in read-heavy contexts — Thai text ("สุนัข · Siberian Husky") is more information-dense than a Dog icon; icons are for scan-heavy tables
4. **hideCloseButton + inline X** — Standard pattern for custom modal headers; avoids absolute positioning alignment issues
5. **Grill-me before code** — 9 atomic questions eliminated all ambiguity; zero wasted implementation cycles
