But I didn't act on any of it until Un pointed it out.

Walking through the decision tree one question at a time — table row actions = NEUTRAL? yes. rounded-full for compact context? yes. outline for view-only? yes. It crystallized the rules cleanly: solid NEUTRAL = my role's state-changing action, outline = other role or view-only, solid+disabled = my role but no permission. That last one was Un's catch, not mine — I had defaulted to outline+disabled which semantically implies "action for another role" rather than "your action but you lack permission." The distinction matters for scannability: a vet scanning the queue should see the dark buttons dimmed on rows they can't pick up, not outline buttons that blend with the "other role" actions.

The bigger lesson is about proactive parity. When a shared component gets new features, I should grep for all consumers and wire them up. Un shouldn't have to notice that call-to-opd was missing chevrons — that's my job. I saved this as a feedback memory, and I need to internalize it beyond just remembering: it's a workflow step, not just a rule.

## Honest Feedback

**Friction 1: Duplicated NextActionIcon across 3 files.** Queue, dashboard, and call-to-opd all have independent copies of the same `NextActionIcon` component with identical logic. When I fixed the disabled button behavior, I had to make the same edit in 3 places. This is a textbook extraction candidate — it should live in `_components/` as a shared component. The duplication is technical debt that made today's fix error-prone.

**Friction 2: I should have run the grill-me proactively.** When Un asked about button styles, I just listed the inconsistency and asked "which direction?" That was lazy. I had the DS right there — I should have checked it first, identified that diagnostic was off-spec, and either fixed it or presented the finding with a recommendation. The DS is source of truth; I shouldn't treat design decisions as open questions when the DS already answered them.

**Friction 3: Callout label UX is limited by plain strings.** The PrototypeGuide label is a `string` type rendered in a `<span>`. I had to add `whitespace-pre-line` just to get newlines working. If the labels get more complex (e.g., clickable links, bold text for HN numbers), the string type will hit a wall. Should consider making `label` accept `React.ReactNode` if this pattern continues.

## Lessons Learned

- **When a shared component gains features, immediately grep all consumers and wire them up.** Don't wait for the user to notice gaps. This is a workflow step, not optional.
- **Check DS before asking design questions.** The DS may already have the answer — present findings, don't ask open questions when the answer exists.
- **Disabled buttons should keep their variant's visual identity.** `solid+disabled` means "your action, no permission." `outline+disabled` means "different role's action, disabled." The distinction is semantic, not cosmetic.
