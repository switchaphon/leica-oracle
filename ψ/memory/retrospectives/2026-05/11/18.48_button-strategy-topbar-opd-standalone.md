## AI Diary

Today was a session of accumulated polish that revealed structural debt. What started as "just add a button color strategy" cascaded into fixing duplicate top bars, rethinking the OPD layout, and learning a lesson about when to share components.

The button strategy work was satisfying — surveying every prototype page, discovering the inconsistencies, and codifying the Neon-agreed rules into a visual decision tree in the DS. The strategy section renders beautifully with its question → variant → class code layout.

The duplicate top bar fix was interesting architecturally. The production NavBar component has real API calls, real session hooks, real cookie management — making it work for prototypes meant enriching mock sessions and handling API failure gracefully. The first attempt (catch-based fallback) failed because the branch API returned empty arrays instead of throwing. A reminder that "failure" in code isn't always what you expect.

The OPD standalone layout decision was the most design-impactful. Moving from sidebar+navbar to a clean breadcrumb bar changes the entire clinical workflow feel — it's now a focused exam room, not a page in a portal. Un pushed me to think about the navigation journey (new tab vs back button vs done-action), and we landed on new tab + done-action — the same pattern Epic uses.

But the lesson that stung was the shared component miss. I built OpdTopBar inline in one page, committed it, and only when Un pointed out order-lab was missing it did I extract. The feedback was sharp and fair: "why not shared from the start?" This is the kind of forward-thinking I should do automatically — check sibling routes before placing UI elements. I saved this as a memory to not repeat it.

## Honest Feedback

**1. Vercel plugin hooks are noisy.** Every time I read a file under `app/`, the hook fires suggestions for `next-cache-components` and `nextjs` skills that are completely irrelevant to prototype UI work. I had to ignore these dozens of times. The pattern matching is too broad — `app/**` catches everything.

**2. The NavBar modification felt fragile.** Adding prototype-specific logic to a production component (`window.location.pathname.startsWith('/prototype')`) is a code smell. A cleaner architecture would be a prototype-specific NavBar that wraps or replaces the production one. But Un wanted minimal changes, and this was the pragmatic choice.

**3. Context window pressure from large git diffs.** The `git diff --stat HEAD~5` output was enormous (400+ files, mostly .playwright-mcp artifacts from past sessions). This eats context for no benefit. Need to consider limiting the diff scope or filtering out noise directories.

## Lessons Learned

1. **Check sibling routes before placing shared UI** — When a component applies to a path prefix (e.g., all `/opd/*` pages), put it in a layout from the start, not inline in one page.
2. **API "failure" isn't always an exception** — `fetchBranchList` returned an empty array (success) instead of throwing (failure). The `.catch` fallback never fired. Always consider what "no data" looks like vs actual errors.
3. **Layout ownership > per-page imports** — Next.js layouts are the right place for persistent UI elements shared across routes. They prevent the "forgot to add it to the new page" problem.
