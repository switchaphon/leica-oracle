## AI Diary

This session was a masterclass in how grilling and documentation interleave. We started with what seemed like a clean continuation — 4 tasks from a handover — but Un immediately spotted gaps and inconsistencies that led to a full /grill-me session. The grill was tight: 9 questions, each building on the last. The OPD terminal state question (COMPLETED vs CLOSED vs DISCHARGED) was the kind of naming decision that seems trivial but propagates everywhere. Getting it right once means never having to rename across 6 docs later.

The most architecturally interesting decision was making Queue truly generic through the "เจ้าของเคส" (case owner) concept. Queue doesn't care if it's a vet or grooming staff — it just knows "the assigned person." This lets IN_SERVICE be a universal wrapper state that spawns any service entity 1:1. It's elegant because it means adding a new service type (say, boarding) requires zero Queue state machine changes.

What surprised me was how much visual alignment matters in documentation. Un caught that the SOAP section items were all purple despite representing different entities, and that mixing downstream flow with side entities in the Connection section created confusion. These aren't code bugs — they're comprehension bugs. A reader seeing all-purple items won't unconsciously map them to the blue/purple/green entity cards. Once we added entity-colored borders, the visual continuity across sections 3→4→5 became immediately scannable.

The diagnostic block placement correction was important: it's not "always in Objective" — it depends on which CTA triggered it. "สั่งตรวจเพิ่ม" from Objective creates a wait=true block there; "สั่งตรวจล่วงหน้า" from Plan creates a wait=false block there. Slash commands default by section but let the user toggle. This is the kind of nuance that gets lost if you document it wrong the first time.

## Honest Feedback

**1. Too many small commits.** 12 commits on the vet app in one session, several being single-line fixes (footer CTA label, SVG label text). These should have been batched. Each commit triggered the full feedback loop of "check → fix → commit → Un spots another thing." A "review pass" before committing would have caught the footer CTA split, the relative link issue, and the color mismatch in one go.

**2. Reading the full file before writing is still a gap.** When I wrote the overall-flow.html rewrite, I had to read it in two pages due to truncation. This fragmented view means I sometimes miss context that spans across the read boundary. For 1200+ line HTML files, I should grep for specific sections rather than reading linearly.

**3. Grill decisions should be tracked as a batch.** After the /grill-me produced 8 decisions, I applied them one-by-one with separate commits instead of planning the full set of edits upfront. This led to follow-up corrections (the footer CTA being combined then split, the entity colors being missed). A "decision → edit plan → single pass → commit" workflow would be cleaner.

## Lessons Learned

- **Visual color continuity across doc sections matters**: when section 3 defines entity colors, sections 4 and 5 must use the same palette. Readers scan colors before reading text.
- **Queue genericity through "case owner" abstraction**: keeping Queue agnostic to service type (vet vs staff) by using a generic "เจ้าของเคส" concept means the state machine never changes when new service types are added.
- **Diagnostic block placement is CTA-dependent, not section-fixed**: the block appears in whichever SOAP section the CTA belongs to, not always Objective.
