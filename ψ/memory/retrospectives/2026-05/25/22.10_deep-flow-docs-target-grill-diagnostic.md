## Key Architecture Decisions

1. **Queue as service-type-agnostic wrapper** — IN_SERVICE spawns any service entity 1:1; "เจ้าของเคส" (case owner) resolves per service_type. Adding new services (boarding, daycare) requires zero Queue state machine changes.

2. **Diagnostic Order data model** — `source_opd_id` (always set), `target_opd_id` (NULL for advance orders), `wait_for_result` (boolean). Auto-attach queries: `WHERE target_opd_id IS NULL AND pet_id = ? AND state = 'COMPLETED'`.

3. **Target-only documentation** — All 6 flow docs now document what's being built. No more As-Is/Target split confusion.

4. **Visual color continuity** — Entity colors defined in Connection section must propagate through SOAP, Timeline, and Transitions. Colors are a comprehension layer parallel to text.

5. **CTA-dependent diagnostic placement** — Block appears in whichever SOAP section the CTA belongs to (Objective → wait=true, Plan → wait=false). Slash commands default by section, toggle in modal.

The auto-attach mechanism for advance diagnostic orders was the intellectual highlight. The problem is elegant: a vet orders a test during Plan ("สั่งตรวจล่วงหน้า"), the patient goes home, the lab completes the test days later, then the patient returns for a follow-up. How does the system know which OPD session should receive those results? The answer — `target_opd_id = NULL` until the next OPD is created, then query by pet_id + state + NULL target — is the kind of design that feels inevitable once you see it but wasn't obvious before we wrote it down.

## Honest Feedback (Deep)

**1. Commit granularity vs batch efficiency.** 18 commits in one session is too many. The "fix after fix" pattern (footer CTA: 3 commits, entity colors: 2 commits, relative links: 1 commit) shows that each commit triggered a follow-up discovery. Root cause: committing before doing a full visual review of all affected sections. Solution: after any grill session, make a single "edit plan" pass listing every section that needs changing, then execute all edits before committing. This would have reduced 18 commits to ~8.

**2. Agent 2 failed on large HTML files.** The File Changes Analysis agent hit context limits because diagnostic-flow.html and queue-flow.html are 800+ lines of inline-styled HTML. This is a signal that the docs themselves are getting too large for efficient AI processing. Not a problem for human readers (they scroll), but a friction point for automated analysis. Consider whether the CSS could be extracted to a shared stylesheet to reduce file size.

**3. Mid-session retro was premature.** I wrote a retrospective at 17:59 that covered "3 hours" of work, but the session continued for 4 more hours after that. The retro captured the grill decisions but missed the entire Diagnostic Order Origin work, the timeline card consistency fixes, and the boundary clarification. Deep retros should only happen at actual session end. The mid-session retro is now partially redundant with this one.

## Lessons Learned

1. **Auto-attach via NULL target_opd_id** — Advance diagnostic orders use target_opd_id = NULL + pet_id match to find their recipient OPD. Query at OPD creation time, attach all COMPLETED orders with NULL target. Edge case: multiple orders attach simultaneously.

2. **Action ≠ Visibility** — Cross-boundary entities (Diagnostic viewed from OPD, Invoice viewed from Queue) show state everywhere but only allow actions from their home page. Document both dimensions separately.

3. **Commit after full visual review, not after each fix** — When applying multiple grill decisions, plan all edits first, execute the full set, visually verify cross-section consistency, then commit once.

4. **4-session arc of "plan → discover → correct"** — Flow docs work spans May 18-25 with a consistent pattern: grill to decide, document Target, discover inconsistency, correct. Each session builds on the last.
