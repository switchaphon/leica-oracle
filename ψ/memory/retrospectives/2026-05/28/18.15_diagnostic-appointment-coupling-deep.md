## Architecture Decisions

1. **Three-Field Reference Pattern**: `source_visit_no` (origin trace, nullable) + `target_visit_no` (result routing, nullable) + `appointment_no` (lifecycle coupling, nullable) — each field serves a distinct purpose, none can substitute for another.

2. **Mirror State Names**: Diagnostic Order gains `NO_SHOW` state matching Appointment `NO_SHOW` — staff understands cascade origin instantly without learning new vocabulary.

3. **State-Dependent Cascade**: DRAFT/PENDING orders auto-cascade; IN_PROGRESS+ orders get flagged for staff review — work already started/completed cannot be auto-cancelled.

4. **Check-in Routing (Option B)**: Advance diagnostic orders → pet goes to Lab/X-ray before exam room. Vet already decided; no re-confirmation needed.

## Honest Feedback

1. **Over-engineering before checking existing state.** I proposed `ordered_by_staff_id` as a new field when it already existed in the UI. This pattern keeps recurring — I design solutions without first auditing what's already built. The fix is simple: before proposing any data model change, read the existing mock data and UI components to see what fields are already there.

2. **Inventing vocabulary when reusing is better.** SUSPENDED vs NO_SHOW is the example here, but the broader issue is my tendency to create new abstractions when the domain already has clear terms. In a Thai vet clinic, staff knows "ไม่มาตามนัด" from appointments — using the same term for diagnostic orders is zero cognitive load. New terms create training debt.

3. **Changelog scope creep.** What started as "update diagnostic-flow.html" expanded to: overall-flow.html sync → CHANGELOG update → per-page changelogs for flow entries → per-page changelogs for ALL entries → fix hardcoded header. Each step was individually correct but the total was ~2 hours of changelog work. Should have asked Un upfront "how far do you want me to go?" instead of discovering scope incrementally.

## Lessons Learned

1. **Mirror state names across lifecycle-coupled entities** — don't invent new vocabulary when the source entity's term is already understood by staff
2. **Three-field reference pattern** — origin trace (source), result routing (target), lifecycle coupling (appointment) are distinct concerns requiring separate nullable fields
3. **Audit existing UI before proposing data model changes** — the field might already exist
4. **Good grill = mechanical implementation** — when all ambiguity is resolved in design, code changes are just translation
