## AI Diary

This session crystallized something I'd been circling for days: the gap between having design decisions and making them visible. DESIGN_SYSTEM.md was thorough — 647 lines of tokens, patterns, rules — but when the user said "I can't see what it looks like," that was the real insight. A markdown file with hex codes is not a design system. A design system is what you can point at in a browser and say "this pink, this radius, this weight — yes or no?"

Building the interactive doc site felt like the right answer precisely because it lives in the same Tailwind config as production. There's no translation layer. What you see IS what the tokens produce. The Button playground with HSL sliders was the proof-of-concept that made the user go "ok I see now." That's the moment I should optimize for — not completeness of documentation, but the click where understanding transfers.

The sidebar debugging was instructive too. First attempt: move to top-level `/design-system` → hit NextAuth redirect (no mock session). Second attempt: nested layout → doesn't override parent in App Router. Third attempt: useEffect hiding DOM elements → works but feels hacky. Sometimes the pragmatic hack is the right answer when the framework doesn't offer a clean escape hatch.

The most important outcome wasn't any code — it was the user explicitly stating: "Design system is source of truth. No one-off styles. New variants go there first." That's a process decision that will shape every future session. I recorded it in auto-memory immediately.

## Honest Feedback

1. **Pixel's brand direction was disconnected from codebase reality.** Pixel proposed "bridge orange + deep teal" as primary colors, which completely ignored that the codebase already uses `#E5007D` as the `brand` token in tailwind.config.ts. Neon got it right because it actually read the config file. Lesson: always include current codebase state in specialist briefs. A "what would be ideal" proposal is useless if it ignores what's already shipped.

2. **The sidebar hiding is a hack.** Using `useEffect` to `display: none` on `#drawer-menu-wrapper` is fragile — if the parent layout changes those IDs, it breaks silently. The proper solution would be a route group or a layout prop, but Next.js App Router doesn't support opting out of parent layouts. I accepted the hack because the alternative (complex routing restructure) would be over-engineering for a prototype page. But it should be flagged if the layout ever changes.

3. **Three navigation attempts before landing.** I went through 3 failed approaches (top-level route → auth redirect, nested layout → doesn't override parent, route group → still inherits) before landing on the useEffect hack. I should have thought through the Next.js layout chain BEFORE moving files. Each failed attempt was a context switch that cost the user patience.

## Lessons Learned

1. **A design system doc isn't real until it renders.** Markdown spec → "I can't see it." Interactive page → "ok I see now." The transfer of understanding happens through rendered pixels, not written words.
2. **Next.js App Router layouts are additive, never replaceable.** You cannot opt out of a parent layout. Accept it and work within the constraint.
3. **Design system as process rule > design system as document.** The user's explicit "no one-off styles" declaration is more valuable than the 647-line spec. Process shapes behavior; documents inform it.
