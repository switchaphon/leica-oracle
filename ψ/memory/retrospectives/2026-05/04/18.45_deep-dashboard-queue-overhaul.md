## AI Diary

This was the session where I learned what "convention" actually means in a codebase. Not guidelines, not suggestions — contracts. When the user said "มันคือหน้าเดียวกันเลยหนิ" I felt the weight of that. Every deviation I'd introduced — a different column name, a different icon, a different color — was a breach of contract that someone would have to find and fix later.

The cascade that followed was instructive. Fixing one inconsistency revealed three more. Changing `สัตว์` to `สัตว์เลี้ยง` led to checking every column header. Fixing `ArrowDownUp` led to checking every import. Fixing `#00001` to `HN66-10-001` led to updating 18 mock data entries. Each fix was small, but the aggregate cost of not doing it right the first time was enormous.

The Neon and Pixel consultations were genuinely valuable — they converged independently on the same color hierarchy, which gave me confidence the framework is right. The icon action button design was satisfying: mapping queue lifecycle states to physical-direction icons (DoorOpen = enter room, CreditCard = go to cashier, Pill = dispense medication) creates a visual language that staff can learn once and read forever.

The biggest win was updating DESIGN_SYSTEM.md with sections 11-14. These aren't just documentation — they're the law. Next time I build a page, I have no excuse for deviating. The terminology table in section 14.1 is a checklist I should read before writing any table cell. The icon mapping in section 12.1 is a lookup table I should reference before adding any action button.

I'm leaving this session with a genuine understanding: read first, clone second, modify third. And when you establish a convention, write it down immediately so it survives across sessions and context windows.

## Honest Feedback

**Friction 1: I still don't read existing code thoroughly enough.** The queue page fiasco was the second time (after the sidebar incident). The memory was there, the convention was there, but I skipped step one. The cost was ~30 minutes of rework plus user frustration. I need a physical checklist: "Did I read the most similar existing page? What are its column headers? What shared components does it use?"

**Friction 2: Responsive behavior requires real-browser testing.** Playwright captures pre-hydration state, so `useMediaQuery` always shows `false`. I spent multiple rounds of "it should work" / "it doesn't" / "try in browser" before accepting this. Should have warned the user upfront and only verified via their screenshots.

**Friction 3: Design system documentation should happen DURING the work, not after.** I updated DESIGN_SYSTEM.md at the very end of a 4.5-hour session. If I'd written section 12 (icon buttons) right after implementing them, the propagation to queue/pickup would have been guided by the spec rather than by memory of what I'd done hours earlier. Write the spec, then implement from it.

## Lessons Learned

1. **Conventions are contracts, not suggestions** — one deviation creates cascading inconsistency across pages
2. **Clone, don't create** — literally copy the file, then modify business logic only
3. **Document conventions immediately** — DESIGN_SYSTEM.md sections 11-14 prevent future deviations
4. **Responsive breakpoints must reflect container width** — 1600px for 70% column, 1280px for full-width
5. **`line-clamp` needs inline styles in `<td>`** — Tailwind class doesn't work, use `-webkit-box` inline
6. **Icon buttons need lifecycle context** — status badge provides the label, icon provides the direction
