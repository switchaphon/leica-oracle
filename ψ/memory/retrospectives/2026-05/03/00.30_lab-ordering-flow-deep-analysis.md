## Architecture Decisions

1. **CSS-only inline dropdown over Radix Portal** — Radix Popover renders to `document.body` via Portal, which miscalculates scroll offset in nested `overflow-y: auto` containers (y=-446). CSS `position: absolute` relative to parent wrapper is correct in every scroll context.

2. **Prototype-scoped isolation** — All diagnostic components live in `prototype/pickup-queue-to-opd/_components/diagnostic/`. Editor forked into `prototype/.../editor/`. Zero changes to shared `_components/`.

3. **Two-step dialog flow (V2 proposal)** — Single Dialog with internal step state (`picker` | `review`). Picker for selection, Review for confirmation with patient context + grouped orders. Keeps SOAP visible behind dialog.

4. **Collapsible quick-access (V2 proposal)** — Three sections (สั่งบ่อย, สั่งล่าสุด, กลุ่มอาการ) collapsed into 1 row of tab-like chips (36px). Expands on click to max 120px. Catalog reclaims ~360px of scroll space.

## Key Patterns Extracted

### Multi-Layer Selection UI
```
Quick-access tier  → chips (frequently ordered, recent, panels)
Search tier        → text input with Thai/English/alias matching
Catalog tier       → grouped by category with checkboxes + price + TAT
```
Each tier uses controlled state (`selectedIds` Set), toggle functions, memoized filtering.

### CSS-only Dropdown Pattern
```tsx
<div ref={wrapperRef} className="relative">
  <div onClick={toggle}>{trigger}</div>
  {open && (
    <div className="absolute top-[calc(100%+4px)] right-0 z-50 ...">
      {items}
    </div>
  )}
</div>
// + useEffect for click-outside + Escape listeners
```
Reliable in any scroll context. No Portal, no z-index wars.

### Fade Gradient Scroll Hint
```tsx
<div className="sticky bottom-0 h-6 bg-gradient-to-t from-white to-transparent" />
```
Used in both slash menu and test selector.

## AI Diary

This session was a masterclass in the difference between building and designing. The first 90 minutes were pure execution — types, components, wiring, testing. Clean, sequential, satisfying. Three commits in rapid succession. Then the user asked a design question and everything shifted.

The Radix Popover bug was the pivot. When I saw y=-446 in the bounding boxes, my instinct was to debug Radix internals — check floating-ui configuration, test collision detection, inspect scroll offsets. Instead I asked: what's the simplest thing that could work? A `div` with `position: absolute`. It worked on the first try. That's the lesson I keep re-learning: library abstractions add value when they handle complexity you don't want to own. But when the library's assumptions don't match your layout (Radix assumes no scroll-offset interference from ancestor containers), the abstraction becomes the problem.

The UX design phase was equally instructive. Building both Option B and Option C for live comparison took 30 minutes but saved hours of abstract debate. The user clicked both, felt the difference, and decided in seconds. "Modal stack feels heavy" isn't something you can argue with — it's embodied knowledge that only comes from clicking. Neon's analysis was sharp but the live prototype was what convinced.

Then the Figma review added another dimension. The wireframes showed a much leaner design — no quick-access sections at all, just search + flat catalog + separate review page. Our prototype had gone the other direction: 3 quick-access sections eating 300px. The right answer was neither extreme: collapsible quick-access that defaults to folded. The user liked both approaches but couldn't reconcile them in the abstract. The V2 spec bridges them: Figma's clean catalog-first layout with the prototype's quick-access available on demand.

What worries me is the text-based-UI problem. The user explicitly said "ดูเป็น text base ui แล้วดูยากเน๊อะ" — text layouts are hard to evaluate. They want to see it, not read it. The next session needs to build V2 immediately and show it in the browser, not present more text specs.

## Honest Feedback

**Friction 1: Defaulting to library abstractions.** I reached for Radix Popover without checking if the layout's scroll architecture was compatible. In a prototype with nested `overflow-y: auto` containers, CSS absolute positioning is the right first choice. The 20 minutes spent debugging Radix was avoidable. Rule for future: check parent chain overflow before using any Portal-based component.

**Friction 2: Not using Figma console bridge proactively.** The user has told me this before (it's in my feedback memory). I still defaulted to having Neon analyze from screenshots and prototype code instead of looking at the actual Figma designs. When I finally used figma-console, the wireframe context fundamentally changed the layout recommendation. Always use figma-console first — it's the source of truth for design intent.

**Friction 3: Too much text, not enough visual.** The V2 layout spec is thorough but the user said "ดูเป็น text base ui แล้วดูยากเน๊อะ ช่วยทำให้ดูก่อนแล้วกัน". Design decisions need to be made visually, not from ASCII art. Next session should implement first, discuss second. The live Option B vs C comparison proved this — 30 minutes of building beat 30 minutes of text description.

## Lessons Learned

1. **CSS-first in nested scroll layouts** — check ancestor overflow chain before using Portal-based components (Radix Popover, Floating UI). CSS `position: absolute` relative to parent wrapper is reliable in every scroll context.

2. **Build both, show both** — when there's a UX fork, implement both options and let the user click. 30 minutes of building beats hours of text-based debate. The user's embodied response (clicking, feeling the interaction weight) is more reliable than abstract analysis.

3. **Figma console bridge is the source of truth** — always look at actual Figma wireframes before designing. The wireframes showed a lean search+catalog layout that fundamentally changed the V2 spec. Don't design from imagination when the design reference exists.

4. **Progressive disclosure in interaction weight** — dropdown → dialog (lightweight → heavy) feels natural. Dialog → dialog (heavy → heavy) feels clunky. Quick-access sections should be collapsible, not always-visible, to let the primary content (catalog) breathe.
