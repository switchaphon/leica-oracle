## Key Architecture Decisions

1. **SOT hex overrides RTD Figma hex** — when SOT and Figma conflict on hex values, SOT wins. SOT is ground truth for data, RTD for visual design.

2. **StatusBadge 2-tone outline** — 3-tone (filled bg) demoted to 2-tone (transparent bg) so status badges are visually subordinate to ServiceTypePill (3-tone + icon). Clear visual hierarchy: service type = primary identity, status = secondary context.

3. **Prototype-local FilterDropdown** — couldn't modify shared production component, so created `/prototype/_components/FilterDropdown.tsx` with proper indeterminate state (pink box + white Minus icon instead of hacky -ml-5 overlay).

4. **Dynamic KPI cards** — replaced static `TODAY_SUMMARY` with computed values from `QUEUE_TABLE_ITEMS`. Numbers now sync with the table below. EMERGENCY/REFERRAL lumped into "วอร์คอิน" count.

5. **Sectioned dropdown (Type E)** — new dropdown archetype with SectionHeading, search, per-item counts. Used for "บริการทั้งหมด" in dashboard. Added to DS as fifth dropdown type.

But the moment I changed Specialized Medicine from red to pink, the visual hierarchy question surfaced: if StatusBadge and ServiceTypePill both use the same 3-tone pattern, how does the user tell which is the service identity and which is the workflow state? ## Honest Feedback

**Friction 1: Dev server killed mid-session.** When I ran `pkill playwright`, it killed the dev server too. Restarting on a different port (3003 instead of 3002) caused confusion — the user was checking port 3002 and seeing an unstyled page. Lesson: never blanket-kill processes; be surgical.

**Friction 2: DS_MARKDOWN inline string is a maintenance nightmare.** The design system page has a ~770-line template literal (`DS_MARKDOWN`) that must stay in sync with the actual `DESIGN.md` file AND the rendered JSX sections. I updated DESIGN.md, updated the rendered components, but forgot the inline DS_MARKDOWN until the user asked about "View .md" and "Download .md". Three sources of truth for what should be one. This will drift again.

**Friction 3: Vercel plugin skill triggers.** Every file read/write under `app/` triggered Vercel skill injection demands (next-cache-components, nextjs, verification, react-best-practices). None were relevant — this is a fully client-rendered prototype with no Server Components. The noise was constant and I had to ignore each one, adding cognitive overhead to every tool call.

## Lessons Learned

1. **Visual hierarchy via badge weight** — 3-tone (filled) for primary identity, 2-tone (outline) for secondary context. The bg-fill is what makes a badge "pop." Removing it creates instant visual subordination without changing the color system.

2. **SOT alignment is always bigger than you think** — a 3-hex-value change touched 16 files. Canonical labels, mock data, design system showcase, inline markdown, filter dropdowns, legend dots, KPI card breakdowns — every surface that renders the concept needs updating.

3. **Indeterminate checkbox needs explicit rendering** — shadcn's Checkbox component doesn't handle `checked="indeterminate"` properly (renders Check icon in black, not Minus in white). Build the three states explicitly: unchecked Checkbox, checked Checkbox, custom button with Minus icon.
