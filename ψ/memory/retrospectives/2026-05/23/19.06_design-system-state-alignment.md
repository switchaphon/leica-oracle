## AI Diary

This session was a masterclass in pattern propagation — taking a well-established template (appointment-flow.html, which we polished over several sessions) and systematically applying its conventions to the queue system. What struck me was how revealing this process is. When I rebuilt the queue's Service Types section to match appointment's pattern, I caught the FOLLOW_UP drift — it had been listed as a top-level service type alongside GENERAL_MEDICINE, when it's actually a subtype. This is exactly the kind of inconsistency that documentation-as-architecture-tool catches.

The advisor guidance was valuable here — especially the warning about Circle icon removal being a "deviation-from-SOT fix dressed up as pattern application." That's a fair observation. The SOT says StatusBadge is 2-tone outline without icon, and the queue section had been adding filled Circle dots. Removing them is correct per SOT but it's worth calling out rather than burying.

The `<a` tag break was my fault — a careless edit that split `<a` into `<` on one line and the attributes on the next. These kinds of JSX split errors are easy to make when inserting content before an element. I should be more careful with edit boundaries in JSX.

The symlink solution for serving docs via the dev server was pragmatic. The HTML docs live in `src/app/prototype/docs/` which Next.js App Router doesn't serve, but `public/` is served statically. A symlink bridges both worlds without duplicating files.

## Honest Feedback

1. **Edit boundary precision in JSX**: The `<a` tag break happened because my edit replacement ended with `<` instead of `<a`. In JSX, splitting a tag name across an edit boundary is a silent bomb — it compiles to nothing useful and the error message ("Expected '</', got '='") is confusing. I need to always include the full tag name when editing around JSX elements, never split at the `<` character.

2. **Color hex verification**: I mapped Tailwind classes to hex values from memory (e.g., `gray-500 = #6B7280`, `indigo-600 = #4F46E5`). These are standard Tailwind values and likely correct, but I didn't verify against the actual Tailwind config. For a design system SOT, this should be verified — even one wrong hex value creates a source of confusion for devs.

3. **State model inconsistency between queue-flow.html and design-system**: The queue-flow.html State Machine SVG still uses the old filled badge colors (yellow bg for CHECKED, orange bg for WAITING, etc.) while the State Badges section now shows 2-tone outline (transparent bg). These represent different visual contexts (state machine diagram vs UI badge), but a dev looking at both in the same document might be confused about which style to implement. The SVG is showing lifecycle flow (fine with filled), but the disconnect should be noted somewhere.

## Lessons Learned

Pattern propagation across documents reveals SOT drift that's invisible when documents are written independently. The FOLLOW_UP service type error existed since the queue-flow.html was first created — it took applying the appointment template to surface it.
