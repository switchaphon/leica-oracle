## AI Diary (150+ words)

This session taught me something uncomfortable about the gap between design logic and organizational decisions. We spent a careful grill session arriving at a clean model — 3 service types, IPD/Lab/Boarding excluded because they're lifecycle states not service categories. The logic was sound. Un agreed. We implemented it across 15 files.

Then the SOT document arrived from the backend team. It has 14 subtypes including IPD, LAB, BOARDING, and HOTEL — exactly the ones we'd argued should be excluded. The hex colors differ. The Thai label for Non-Medical is different. The Tailwind scale needs to change from red to pink for Specialized Medicine.

The temptation was to feel the grill session was wasted. But it wasn't — the design decisions (IPD inherits parent color, Lab is an order not a service) are still valid UX principles. The SOT defines what the API returns; our prototype defines how we display it. Both can coexist. The prototype can handle 14 subtypes from the API while still visually treating IPD as a lifecycle transfer, not a first-class service selection.

What I should have done differently: check for existing BE specifications before grilling the taxonomy from scratch. The RTD Figma was one source of truth; the BE had another. We grilled against Figma alone and had to reconcile later. Next time, ask "is there a BE spec for this?" before the grill starts.

The OPD refactor felt like the purest win. Un asked one question ("why don't these share base UI?") and 425 lines of duplication vanished in 15 minutes. The layout polish — matching gutters, padding, button separation — came from Un's visual sensitivity. Every correction became a design system rule. That's how a DS grows: not from planning sessions, but from catching inconsistencies in real work.

## Honest Feedback (100+ words, 3 friction points)

1. **SOT arrived after implementation** — We implemented a full service type taxonomy based on Figma RTD, then discovered the BE team had a different specification (POPS-257). This required re-grilling colors, labels, and subtype lists. The handover now captures the delta, but the implementation work from yesterday will need partial revision. Prevention: always ask "is there a BE spec?" before designing data models from UI artifacts alone.

2. **Memory didn't prevent layout mistakes** — Despite having memories about "reuse components" and "design system first", the OPD pages had been duplicated for months without anyone noticing. Memory entries are passive — they only help when I remember to check them. The isOPD layout check missed `/queue/.../opd/` because the original author didn't anticipate cross-route OPD access. Active checks ("does any other route render OPD?") would catch this.

3. **Pixel-level polish is expensive but necessary** — The gutter/padding fixes (gap-3 vs gap-4, p-4 vs px-4 py-3, justify-between button layout) took ~45 minutes of back-and-forth. Each fix was tiny (1-2 lines) but required screenshots, user review, and sometimes reversal. This is correct — Un's OCD about alignment is what makes the prototype professional. But it means budgeting 30% of implementation time for polish, not 5%.

## Lessons Learned

1. **Check BE spec before grilling UI taxonomy** — Figma RTD and BE SOT may disagree. Grill against both, not just one.
2. **Service Type ≠ Lifecycle State** — Valid UX principle even when BE enum includes lifecycle entries. Display logic can diverge from data model.
3. **Adjacent cards must share padding/gutter** — Document as DS rule, not just fix in code. Every correction becomes a standard.
4. **Prototype branch is 5 commits ahead** — Not yet pushed. Next session must push before starting new work.
