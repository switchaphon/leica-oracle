## AI Diary

This session was a satisfying exercise in surgical debugging. The "command too long" error looked trivial at first — I assumed it was the inbox drain blowing past ARG_MAX. But macOS ARG_MAX is 1MB, and the inbox was only 16KB. That mismatch sent me down the right path.

The real detective work was tracing through four layers of abstraction: `drainWakeInbox` → `sendWakeCommandAndPrompt` → `sendPromptViaTmux` → `tmux.run("send-keys")` → `hostExec("bash -c ...")`. Each layer had its own quoting, buffering, and escaping logic. What made it click was finding `isClaudeLikeEngine`: when `engine` is `undefined` (the default wake case with no `--engine` flag), it returns `false` because of a guard on line 31 — `if (!name) return false`. This was written to handle the "engine unknown" case conservatively, but it accidentally routed ALL default wakes through the `send-keys` path instead of the buffer-safe `load-buffer` path.

The fix was simple — six lines — but the diagnosis took 80% of the time. I read through wake-cmd.ts (1300 lines), tmux-class.ts, ssh.ts, is-claude-like.ts, and wake-inbox-drain.ts to understand the full pipeline. The code is well-structured but the engine detection edge case was non-obvious.

One concern: this is a local patch to an installed npm package. It'll be lost on the next `maw-js` update. Worth upstreaming.

## Honest Feedback

1. **Auto-classifier blocked the Discord bot launch** — The `--dangerously-skip-permissions` flag triggered a safety block even though this is a routine operation the user does constantly. For known oracle bot patterns, this should be pre-approved in settings. I had to punt the command to the user to run manually, which breaks the flow.

2. **Deep code tracing is context-expensive** — I read ~2000 lines of maw-js source across 6 files to find a 1-line root cause. The code is well-organized but the indirection (sendWakeCommandAndPrompt → sendPromptViaTmux → tmux.run → hostExec) makes it hard to reason about what actually happens without reading everything. A comment like "NOTE: send-keys has a ~4KB arg limit" at the call site would have saved 10 minutes.

3. **Installed npm package patching is fragile** — Editing `node_modules` directly is a known antipattern. The fix works now but will be silently reverted on any `npm update` or `bun install`. There should be a `patches/` directory or upstream PR workflow to make these durable.

## Lessons Learned

- `isClaudeLikeEngine(undefined)` returns `false` — default wakes without explicit `--engine` flag bypass Claude-specific optimizations like `-p` prompt passing and use raw `tmux send-keys` instead
- `tmux send-keys` has a practical argument length limit that `load-buffer` (stdin-based) does not
- When debugging "too long" errors, check which specific command in the chain is too long — the limit may not be where you expect (it wasn't ARG_MAX, it was tmux send-keys)
