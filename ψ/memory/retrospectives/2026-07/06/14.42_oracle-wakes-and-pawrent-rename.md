## AI Diary

Second rename in three days. The pattern from pops-clinic → pops-vet paid off immediately — I had the 9-surface checklist memorized, the gotchas documented, and the execution was mechanical rather than exploratory. That's the whole point of the learning system: first time costs hours of discovery, second time costs minutes of execution.

The inbox bloat issue was interesting. Three oracles independently accumulated enough inbox messages to exceed the CLI argument limit. The root cause isn't maw draining messages — that's correct behavior. The root cause is that nobody cleaned inbox after reading. Oracle sessions read the messages but leave the files in place, so they accumulate across sessions until the combined content exceeds the shell argument buffer. This is a systemic issue — it'll hit every oracle eventually. The fix (archive old messages) works but it's manual. A proper fix would be maw marking messages as read after drain, or the oracle archiving on wake.

The rename itself was clean. No worktrees to repair, no cross-oracle sync_peers to update, no live sessions to coordinate around. The hardest part was the sweep — finding every CLAUDE.md across the fleet that still said "pawrent-oracle". I found refs in relay-oracle and rpro-saas-oracle that still had the *previous* stale name (pops-clinic-oracle), which means the first rename didn't sweep wide enough. I fixed both while I was there.

Un's request to add a root-level symlink at `pops-pet/ψ` was smart — it means any Claude session opened at the pops-pet group root can see the oracle brain, not just sessions opened inside `pawrent/frontend/app/`.

## Honest Feedback

1. **Inbox accumulation is a time bomb.** Every oracle will eventually hit this. I should propose a convention: after an oracle session reads its inbox, archive processed messages. Or maw should auto-archive after drain. Without this, every `maw wake` is a ticking clock toward "command too long."

2. **Rename sweep was incomplete last time.** The pops-clinic → pops-vet rename on Jul 4 missed relay-oracle and rpro-saas-oracle. I only caught it because I did a fleet-wide grep this time. The 9-surface pattern should explicitly include "grep ALL oracle CLAUDE.md files, not just the ones you remember."

3. **The Vercel plugin kept injecting irrelevant skill suggestions.** Reading `.env` triggered a Vercel env-vars skill. Reading `README.md` triggered a bootstrap skill. The user's prompt mentioning "ppr" triggered next-cache-components. None of these were relevant — this is an oracle infrastructure repo, not a web app. The noise is tolerable but it's friction.

## Lessons Learned

1. **Inbox archive convention needed**: Oracle inbox messages should be archived after processing to prevent maw wake failures. Systematic issue across the fleet — not a one-off.
2. **Rename sweep must be fleet-wide**: `grep -rl "old-name" ~/ghq/github.com/switchaphon/*/CLAUDE.md` should be step 7.5 in the 9-surface pattern.
