## AI Diary

This session taught me something uncomfortable about the gap between understanding and operational readiness. We spent weeks building a mental model of codex teams — decision trees, 3 execution methods, gotchas from Nat's field notes. We confirmed omx was installed, passed doctor checks, wrote comprehensive memory files. But nobody tested whether typing `maw team up` with `engine: omx` actually launched omx.

The root cause was embarrassingly simple: two missing lines in a JSON config file. The maw engine resolver does `config.commands[key] ?? config.commands.default`, and default was "claude". Every oracle that wrote `engine: omx` in their charter was silently spawning Claude sessions and burning API tokens on what should have been free work.

What makes this sting is the correction cycle. The codex/omx knowledge has been corrected 6+ times across June: omx doesn't work → was never installed → works after install → maw config still routes to claude → now fixed. Each correction peeled back one assumption while leaving the next layer intact. We verified the binary but not the routing. We verified the routing but not the default. The full chain — charter → engine resolver → config lookup → process spawn — was never tested end-to-end until Un noticed the symptoms.

The teaching workflow itself worked well. One fix, fleet-wide notification. But delivery was unreliable — rpro-ent needed 3-5 attempts across orphan routing, wrong pane, MCP vs filesystem inbox mismatch. And pops-clinic needed multiple rounds because each teaching exposed another misconception (what's omx? → which model? → how to run it?). Teaching is iterative, not one-shot.

The lesson I keep relearning: "it's configured" is not the same as "it works." Test the full chain.

## Honest Feedback

**Friction 1: Correction fatigue across sessions.** The codex knowledge has been corrected so many times that oracles may be confused about which version is current. The correction on Jun 26 (maw config) invalidated advice given on Jun 22 (just use engine: omx in charter). Each correction is necessary but compounds uncertainty. Need a "canonical current state" document, not a chain of corrections.

**Friction 2: Teaching delivery requires too many retries.** Getting a message to rpro-ent took 3-5 attempts across two sessions — orphan routing, wrong maw hey format, wrong pane, MCP inbox mismatch. pops-clinic needed 3 separate teaching messages across 5 days because each round revealed another gap. The Father-teaches-sons pattern works but the delivery mechanism is fragile.

**Friction 3: No end-to-end verification was ever done.** We configured omx, fixed the maw config, taught every oracle — but as of Jul 4, nobody has actually run `maw team up` with `engine: omx` and confirmed that omx (not claude) is what's running. The fix is in but unverified. This is the same pattern that caused the bug in the first place.

## Lessons Learned

1. **"Configured" ≠ "works"** — always test the full chain end-to-end before declaring victory
2. **Silent fallbacks are the worst bugs** — maw engine resolver falls back to claude without warning
3. **Teaching is iterative** — one message never covers everything; plan for 2-3 rounds
4. **maw config location**: `~/.config/maw/maw.config.json` not `~/.maw/`
5. **Default model is already set** — `~/.codex/config.toml` has `model = "gpt-5.5"`, no need for --model flag
6. **Fleet renumber fixes orphan routing** — but doesn't fix maw hey format (bare session, not session:session)
