## Honest Feedback

**1. PRD publishing muscle memory is wrong for this project.** The /to-prd skill defaults to creating GitHub issues, but pops/vet keeps PRDs as local .md files in `prototype/prp/`. I followed the skill blindly instead of checking project conventions first. This is the kind of error that erodes trust — Un had to correct me on something I should have known. I need to check project conventions before following any skill's default publication path.

**2. Sibling pet placement took 3 iterations.** Pills → rows → compact links. Each time Un said "too tall" or "wrong component" and I rebuilt. The right approach would have been to ask upfront: "How much vertical space can we spend on siblings?" instead of guessing with pills first. I optimized for my assumption of what looks good instead of asking the constraint first.

**3. The grill could have been faster with upfront Figma reads.** Un provided 6 Figma links but I only looked at the inline screenshots. If I had fetched the Figma designs via MCP at the start, some questions (like "does full view have sidebar?") would have been pre-answered. The convention says "always try figma-console MCP before REST API plugin" but for read-only reference images, I should have at least checked what was available.

## Lessons Learned

- **Project conventions override skill defaults** — /to-prd says "publish to issue tracker" but pops/vet keeps PRDs local. Always check where the project stores its artifacts before following any skill's publish step.
- **Ask the constraint before building the UI** — "how much space?" before "what component?". Iterating 3 times on sibling placement was avoidable.
- **Grill decisions compound** — D2 (separate routes) unlocked D5 (context-dependent sibling behavior) which unlocked D8 (both views in app shell). Early structural decisions echo through every subsequent branch.
