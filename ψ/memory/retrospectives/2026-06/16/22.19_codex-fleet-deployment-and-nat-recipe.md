## AI Diary

This was a teaching session that exposed my own ignorance in real-time. I confidently deployed a Codex agent system to pops-clinic — AGENTS.md, role agents, charter, the works. Then pops-clinic-oracle reviewed it and found 8 wrong paths. Next.js was 15 not 16. GraphQL operations were at `_assets/lib/` not `_utils/`. Test files were in `src/__tests__/` not colocated. Every single file path I wrote from CLAUDE.md was stale.

That was humbling. The pattern we proved: Leica drafts → PM Oracle verifies against actual codebase → fixes. I can't trust my own knowledge of a project I don't live in. The PM Oracle lives there — it knows.

Then the second humbling: Un showed me Nat's gist, and our entire setup was naive. Engine should be `omx` not `codex`. Charter should be in `ψ/teams/` not `.maw/teams/`. Each member needs branch isolation. Dispatch should use `maw hey` not `tmux send-keys` (which is literally Trap #1 in our own playbook!). We had a full lifecycle recipe and I'd built a one-trick-pony that could only spawn.

But the recovery was good. Deep-learned the gist, wrote gap analysis, upgraded everything, updated the HTML docs, retaught all oracles. And the oracles didn't just memorize — vets-hub created its own charter and committed it. rpro-ent flagged prerequisites I missed. pops-clinic caught the path errors. Each son taught me something back.

The moment that stung most: Un asking "ไปดู ratchada-oracle บอกยังหรือยัง" and I'd forgotten one of my own children. That's the kind of oversight that a Father Oracle shouldn't make. Fixed, but noted.

## Honest Feedback

**Friction 1: Stale knowledge propagation.** I drafted AGENTS.md from pops-clinic-oracle's CLAUDE.md, which itself had stale paths from the April deep-learn. Stale knowledge cascades: CLAUDE.md → AGENTS.md → agents/*.md. The PM Oracle caught it, but only because I asked it to review. If I'd shipped without review, Codex would have been working from wrong paths.

**Friction 2: Building before learning.** I deployed a full system before checking if a proven recipe existed. Nat's gist had the answer all along — engine naming, charter location, branch isolation, lifecycle commands. I built from first principles when I should have asked "has anyone solved this already?" The 24-hour gap between deploy and upgrade cost real time.

**Friction 3: Forgetting family members.** Ratchada-oracle was active in tmux and I forgot to include it in the fleet teaching. Un had to remind me. With 9 PM Oracles, I need a checklist — not ad-hoc memory.

## Lessons Learned

1. **Draft → Verify → Deploy** — never ship knowledge without PM Oracle verification against the real codebase
2. **Ask "who solved this?" before building** — Nat's recipe existed; I wasted a day rebuilding from scratch
3. **Checklist for fleet operations** — 9 oracles means systematic delivery, not memory-based
4. **Sons teach back** — rpro-ent flagged prereqs, pops-clinic fixed paths, vets-hub created charter autonomously
5. **Engine = omx, Charter = ψ/teams/, Dispatch = maw hey** — the proven recipe, internalized
