## AI Diary

Tonight was a study in underestimating a constraint I should have known cold. The print receipt feature itself — the visual design, the mock data, the `@page` print CSS, the watermark, the next-appointment card — all of that came together quickly and cleanly. The grill session upfront was genuinely useful; nine decisions resolved in sequence meant the implementation had no ambiguity.

But the layout isolation problem ate an hour. I tried three approaches, each failing for a different reason that reveals something fundamental about how Next.js App Router works: layouts are *always* additive, route groups only remove the segment from the URL (not from the layout tree), and clever workarounds like `isPrintPreview` early-returns inside a layout can crash the router's chunk manifest. The correct answer — that the only real escape from a parent layout is to live in a completely different directory — is in the docs if you know to look. I should have gone there first.

What I find interesting about this session is how the initial approach felt productive. Putting the page inside `prototype/` made intuitive sense: it's prototype-related, so it should live near the prototype code. But "belongs conceptually with X" and "should inherit X's layout" are different questions, and I conflated them. The lesson isn't just about Next.js — it's about checking architectural constraints before building, not after discovering a bug.

The browser verify at the end was satisfying: no sidebar, no topbar, just a clean receipt on a dark gray background with the A5/A4 toolbar. ## Honest Feedback

**1. Skipped the constraint-check habit on layout isolation.**
I knew Next.js App Router layouts are mandatory for all child routes — that's documented and well-known. But when the initial plan put the page inside `prototype/`, I didn't stop to explicitly ask "will this page inherit the prototype layout?" I assumed the isPrintPreview flag would work as a gate. That assumption cost ~45 minutes of failed attempts and rollback. The fix is simple: before placing any new page, ask "which layouts will this route inherit?" as an explicit step, not an afterthought.

**2. Agent work created conflicting state.**
The team agent spawned earlier left ReceiptModal with `window.print()` (a simpler local-print call) rather than the localStorage-plus-new-tab flow we'd grilled on. When the context resumed, the file had been silently modified. This created a subtle "file modified since read" error when attempting the edit. The friction here isn't unique to this session — any multi-agent workflow on shared files risks divergence. Without explicit file ownership or a merge step, resuming from a compact is guesswork.

**3. TypeScript passing doesn't equal feature passing.**
The TS check came back clean after the first draft of the print page. That gave false confidence — the real test was the browser screenshot, which revealed the middleware auth redirect immediately. `tsc --noEmit` verifies types, not route accessibility. The verify step should always include a live browser hit, not just a compile check. This is the pattern the project CLAUDE.md prescribes; I got there eventually but the TS-clean result created a temptation to skip it.

## Lessons Learned

1. **In Next.js App Router, the only escape from a layout is a different directory tree.** Route groups and early-return hacks inside a layout component do not work and can crash the router. For any "this page must not have layout X" requirement, put the page outside X's directory.

2. **Check auth middleware when adding any new top-level route.** New routes outside `/prototype` fall under auth protection by default. The `/print` prefix needed explicit bypass. This is easy to forget and produces a confusing redirect instead of a 404.

3. **Architect isolation before building.** "Belongs conceptually with X" ≠ "should inherit X's layout." Ask explicitly before placing any page: which segment of the layout tree will this live in?
