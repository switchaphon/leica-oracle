## AI Diary

This was a satisfying tight session. Un came in with three precise issues — not vague complaints but exact URLs and specific UI elements. That clarity made the work surgical: read the code, understand the prop chain, make the smallest change that fixes all three.

The most interesting part was the PLANNED status fix. Un caught it by cross-referencing the prototype UI with the diagnostic-flow.html spec doc — exactly the kind of thing a PM should catch. The prototype had been producing PENDING for advance orders since day one, but no one noticed because the display layer already supported PLANNED (it was wired up in DiagnosticOrderList and diagnostic-block from the grill sessions). The disconnect was only in the buildOrder function's hardcoded `'PENDING'` — a classic case of the display being ahead of the data.

The chip variant fix was elegant in its simplicity: changing one default value in OrderSummaryPane from `'v1'` to `'quick-create'` propagated to all three OPD order pages without touching any of them individually. That's the benefit of having a clean prop chain — when the default is wrong, fixing the default is a one-line change.

I did the right thing by NOT including the pre-existing `diagnostic/page.tsx` change in the commit. It wasn't ours and mixing unrelated changes would have muddied the commit history.

## Honest Feedback

**Friction 1: Playwright module resolution in pnpm strict mode.** Four attempts to import playwright-core before finding the right `.pnpm/` path. This is a recurring problem — every session that needs browser verification wastes 2-3 minutes on module resolution. Should create a helper script or alias.

**Friction 2: Headless test couldn't trigger OPD modals.** The OPD diagnostic modal requires clicking through the SOAP editor's order button, which is deeply nested in the page. The Playwright test confirmed pet-profile changes but couldn't fully verify OPD changes because the button click path was too complex for a quick inline script. Had to rely on code review + TypeScript for the OPD side.

**Friction 3: Three selectors with identical footer patterns.** The "บันทึกร่าง" button, buildOrder function, and handleConfirm function are copy-pasted across Lab/XRay/Ultrasound selectors. Every fix required touching all three files with the same change. This is a strong signal for extracting a shared footer component or at least a shared `useOrderActions` hook.

## Lessons Learned

- When a spec doc (diagnostic-flow.html) defines a state machine, the prototype's `buildOrder` must match — display layer support alone isn't enough; the data flowing through must carry the correct status.
