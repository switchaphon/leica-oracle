## Honest Feedback

**Friction 1: maw hey format confusion.** I used `09-leica:05-rpro-ent` format for cross-session messaging, which was wrong. The colon separates session from window/member, not session from session. This cost 3-4 failed attempts before discovering `maw hey <session>` is the correct form. The maw documentation should make this clearer, or I should have tested the format before sending important messages.

**Friction 2: rpro-ent inbox reading via MCP.** rpro-ent uses arra-oracle-v3 MCP to read inbox, not direct filesystem reads. So files we dropped in `ψ/inbox/` were invisible to it. This is an architecture mismatch — `maw hey` persists to filesystem inbox, but some oracles read inbox through MCP servers that may have different paths. Need to understand which oracles use which inbox reading method.

**Friction 3: Skill installation location ambiguity.** I initially recommended installing skills to the oracle repo's `.claude/skills/`. Un correctly pointed out that generic skills should go to `~/.claude/skills/` (user-level). The mental model should be: generic tools = user level, project-specific config = repo level. I should apply this principle automatically.

## Lessons Learned

1. **maw hey format**: bare `maw hey <session>` for cross-session, `maw hey <session>:<member>` for within-team only
2. **Skill installation hierarchy**: generic skills → `~/.claude/skills/`, project-specific → repo `.claude/skills/`
3. **Fleet orphan fix**: `maw fleet renumber` resolves slot 00 / orphan routing issues
4. **Trust phase is critical**: pre-trust repos in config.toml before any codex team deployment
5. **Inbox delivery varies by oracle**: some read filesystem, some read via MCP — `maw send-enter` as reliable fallback
