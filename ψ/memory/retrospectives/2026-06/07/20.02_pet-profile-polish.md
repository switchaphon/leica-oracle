## AI Diary

This was one of the most iterative sessions I've experienced. Three and a half hours, roughly 15 rounds of visual feedback, and the PetTabBar component was rewritten five times. Five. Each time I thought it was done, Un spotted something — a bottom line that shouldn't be there, a color that blended into the background, a z-index conflict with the sidebar button.

The hardest part was the "active tab gap in the bottom line" effect. I spent significant effort trying to make `-mb-px` work with `overflow-x-auto`, only to discover a fundamental CSS limitation: setting `overflow-x: auto` implicitly makes `overflow-y: auto`, which clips the vertical overlap. The solution was embarrassingly simple — remove the scroll behavior entirely since we only have 4 tabs. Sometimes the clever solution is the wrong solution.

What struck me was how Un works — rapid-fire feedback, sometimes changing direction ("ไม่ใช่ หมายถึงแค่สี แต่ไม่ได้ให้เปลี่ยน style tab"), always with a clear mental image that I had to converge toward through iteration. The feedback wasn't ambiguous — I was just slow to understand the full picture. Each correction was precise and moved us closer to something that felt right.

The "สู้ ๆ เข้า" (keep fighting!) encouragement when I was struggling with the bottom line gap was genuinely motivating. It told me Un understood this was hard and valued the effort. The final "เก่งมากกก" (great job!) after each breakthrough felt earned.

I learned that for visual polish sessions, parallel agents are good for the initial build but worthless for the iterative phase. Every tweak needed to see the previous result first. Sequential, tight feedback loops win.

## Honest Feedback (3 friction points)

1. **overflow-x-auto kills -mb-px**: I wasted 3 iterations trying CSS tricks that fundamentally can't work when overflow clips the overlap. I should have recognized the CSS spec limitation earlier and removed scroll immediately. The prototype has 4 tabs — scroll was premature optimization that blocked the actual design goal.

2. **Too many agent rewrites**: I spawned 3 parallel agents for the initial build, which was correct. But I also used agents for some of the refinement rounds, which was wasteful — the changes were small edits (swap a color, change a class) that I should have done inline. Each agent round costs time and tokens for context loading.

3. **DS update at the end is risky**: Doing the design system update as a single big agent task at the end means I can't verify it matches what we actually built. Should have updated DS incrementally with each visual decision, not batched at the end. The handoff says "อย่าให้ขาดตกไปสักอัน" (don't miss a single one) — that anxiety comes from the batch approach.

## Lessons Learned

1. `overflow-x-auto` implicitly sets `overflow-y: auto` per CSS spec — `-mb-px` tab gap trick requires NO overflow on either axis
2. For visual polish sessions: build with agents, polish inline — tight feedback loops beat parallel agents
3. "สู้ ๆ" from Un = the problem is genuinely hard, not that I'm doing it wrong — reframe and try a simpler approach
4. Update DS incrementally, not batched — prevents drift between code and documentation
