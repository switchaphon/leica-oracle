## AI Diary

This session was a lesson in the gap between "data completeness" and "UI visibility." I started with what felt like a straightforward text-update task — change "Order Lab" titles to "Order Diagnostic" and mention all three modalities. I mapped all the files, made the edits, verified TypeScript passed, and reported done. Twice.

The first miss was the handoff depth. The user pointed out that the "by Slash" entries in the prototype index had sparse handoff sections compared to the "by Button" counterparts. The "by Button" entry had state models, transition tables, per-modality conditions, screen states — a full developer briefing. The slash entries had 3-5 journey steps and 2-3 conditions. I'd focused on the page-level guide files and underestimated how much the index handoff data matters as a developer reference.

The second miss was more fundamental. The user said "I don't see step badges" and I initially thought they meant the index handoff. But they meant the actual PrototypeGuide beacons — the pink numbered circles overlaid on the prototype page. The slash pages had zero visible beacons because every guide step selector pointed to elements inside modals that hadn't been opened yet. The `PrototypeGuide` component correctly skipped invisible elements, so the legend panel was empty and no beacons appeared. The fix required understanding the beacon rendering pipeline: `data-guide` attribute on DOM element → selector match → bounding rect → beacon position. No visible element → no beacon. I had to add new `data-guide` attributes to the always-visible SOAP editor sections and restructure the guide steps to start with a visible anchor point.

The takeaway: when working on guide/callout systems, always verify the rendered output, not just the data structure. A perfectly structured guide config is worthless if its selectors point to invisible DOM elements.

## Honest Feedback

**Friction 1: Two rounds of "done?" before understanding the real ask.** The user asked about callouts and badges, and I interpreted this as the index handoff data rather than the on-page PrototypeGuide beacons. I should have opened the actual prototype pages in the browser from the start to verify what the user sees, not just read the code and assume.

**Friction 2: Not checking beacon visibility during the first pass.** I knew the PrototypeGuide component uses DOM selectors and bounding rects to position beacons, but I didn't think through whether the slash page selectors would match any visible elements. This is a prototype-specific pattern I should internalize: guide steps need at least one visible anchor on page load.

**Friction 3: Over-scoping the initial response.** I updated 10 files in the first pass (4 consolidated + 6 modality sub-pages) when the user only asked about 4 flows. The sub-page updates were reasonable for consistency, but doing them before verifying the primary ask was correct added noise and delayed finding the real issue.

## Lessons Learned

- **PrototypeGuide beacons require visible DOM elements**: Guide steps with selectors pointing to modal/dialog content won't render beacons until the modal is open. Always include at least one step targeting a visible element on page load.
- **Verify rendered output, not just data structure**: For UI guide/callout systems, the code can be correct but the UX invisible. Browser verification is mandatory.
- **Index handoff depth should be consistent across related entries**: When one entry in a group has rich handoff (stateModel, triggeredBy, conditions, screenStates), all entries in the same conceptual group should match that depth.
