## Honest Feedback

**Friction 1: Revert churn from misunderstanding.** When the user said "panel สีดำ" I immediately assumed VoiceRecordingBar and edited 3 files (bar position, preview rounded corners, page padding). The user meant the PrototypeGuide bottom bar. Three reverts + the correct edit = wasted round-trips. I should have taken a screenshot first and asked "this one?" before touching code.

**Friction 2: Playwright browser lifecycle.** The Playwright MCP kept getting stuck with "browser already in use" errors, requiring `pkill -f mcp-chrome` + retry. This happened 3 times during the session. The workflow of navigate -> screenshot -> click -> screenshot is fragile when the browser state gets corrupted. A more robust reconnection pattern would save real minutes.

**Friction 3: File discovery confusion with rtk.** The `find` command couldn't locate files in `public/` because rtk's filtering excluded them. I spent several commands trying different search patterns before using `stat` directly. Knowing that `public/` is filtered by rtk would have saved the back-and-forth.

## Lessons Learned

- Always screenshot first before editing when the user describes a UI element by appearance ("panel สีดำ") — confirm which element before touching code.
- `public/` files in this repo are invisible to rtk-filtered `find` — use `stat` or `ls` directly.
- Pitch deck tone-matching is faster when you copy the exact CSS from the reference file rather than building from memory.
