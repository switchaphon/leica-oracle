## AI Diary

This session taught me something about the difference between building a feature and proving a feature works. I wired the Tiptap insertion, added toasts, fixed the state machine, built a polished pitch deck -- and then the actual voice recording didn't work on the user's machine because Chrome couldn't reach Google's speech servers. A `network` error. Nothing wrong with the code.

The mock fallback system we built into the PRD turned out to be not just a nice-to-have but the only way to demo the feature. That decision from yesterday's grill session -- "what if Web Speech API fails during demo?" -- was prescient. The mock button is the real star of the pitch.

The revert churn from misidentifying "panel สีดำ" still bugs me. Three files edited, three files reverted, all because I assumed instead of asking. The lesson I wrote about screenshotting before editing is real. But honestly, in a fast-moving prototype session, the instinct to "just fix it" is strong. The discipline is in slowing down for one screenshot when the request is ambiguous.

The pitch deck rewrite was humbling too. My first version had its own design system -- dark hero, grid cards, section dividers. When the user pointed to the OCR pitch as reference, I realized consistency matters more than creativity in a pitch suite. Same CSS tokens, same layout patterns, same Thai-first punchy copy. The user sees a product family, not isolated experiments.

The Web Speech API network issue is a real limitation for this demo. Phase 2 with Whisper API will solve it (no Google server dependency), but for now the mock fallback is the safety net. The error messages I added (Thai, specific per error type) at least tell the user what's wrong instead of silently failing.

## Honest Feedback

**Friction 1: Visual element misidentification.** "Panel สีดำ" could mean VoiceRecordingBar (dark waveform), PrototypeGuide bottom bar (neutral-900), or the dark section of WaveformVisualizer. I guessed wrong and edited 3 files. The fix is always screenshot + confirm, but the temptation to just act is strong in prototype sessions where speed feels valued.

**Friction 2: Web Speech API is a fragile foundation.** The `network` error means Chrome sends audio to Google servers for processing. If the network path to Google is blocked (VPN, corporate firewall, ISP issue), the entire feature silently dies. The error handling I added helps diagnose, but the fundamental dependency on Google's infrastructure is a demo risk. The mock fallback mitigates but doesn't eliminate the "it doesn't work" moment.

**Friction 3: Double file creation for static HTML.** I created the pitch HTML in `src/app/prototype/docs/` first, then discovered it needs to be in `public/prototype/docs/` to be served by Next.js. Had to copy and delete. Should have checked existing patterns (`public/prototype/docs/*.html`) before creating the file.

## Lessons Learned

- Web Speech API `network` error = Chrome can't reach Google speech servers. Not a code bug. Caused by VPN, firewall, or ISP blocking. Mock fallback is essential for demo reliability.
- Screenshot before editing when user describes UI by appearance, not component name.
- Static HTML docs go in `public/prototype/docs/`, not `src/app/prototype/docs/`.
- Pitch deck consistency > creativity: copy the reference CSS, change the content.
