## AI Diary

This was a clean, focused session — the kind I enjoy most. No ambiguity, no design decisions, just ground-truth verification. Read the docs, check the code, fix what's wrong.

What struck me is how many paths were wrong. Leica extracted from CLAUDE.md, which itself was written from a deep-learn snapshot taken 2026-04-28. The project has evolved since — `graphql-operations.ts` moved from `_utils/` to `_assets/lib/`, the auth file moved up one directory, the route group was renamed from some earlier convention to `(routes)`. These aren't mistakes by Leica — they're drift between the snapshot in CLAUDE.md and codebase reality. The deep-learn artifacts are frozen; the code keeps moving.

The most satisfying fix was the SWR keys. chrome.md imported from a non-existent `@/app/_utils/swr-keys` file. The real pattern is hook-local: each hook file exports its own keys (PET_LIST_KEY from pet.ts, OWNER_LIST_KEY from owner.ts). This matters because a Codex agent following the wrong import path would immediately fail and waste its entire context window on a dead end.

The `initGraphQLClient` being async was another subtle catch. The examples showed synchronous usage. A Codex agent copying that pattern would get a Promise where it expected a client object — a confusing runtime error with no compile-time signal.

Eight fixes total. None were conceptual errors — every description of the auth model, RBAC dual-system, and architecture constraints was accurate. The problems were all path references. This tells me the knowledge transfer was solid; only the filesystem map was stale.

## Lessons Learned

- Codex agent docs derived from CLAUDE.md inherit CLAUDE.md's staleness — fix the source, not just the copy
- Always verify file paths with `find`/`ls` before documenting them — paths drift faster than concepts
- Hook-local SWR keys (no centralized key file) is a deliberate pattern that must be documented accurately for stateless agents
