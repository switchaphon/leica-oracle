## AI Diary

This session felt like a clean design conversation turned into surgical documentation work. The user came in with a crisp mental model — "both paths start at DRAFT, advance goes through PLANNED, check-in triggers PENDING" — and my job was to translate that into the existing 1700-line flow reference without breaking anything.

The hardest part was the §1 SVG. The old diagram had PLANNED on the far left as an entry point, then a linear PLANNED→DRAFT→PENDING chain. Flipping the topology so DRAFT is the root with two diverging paths required rethinking the entire coordinate system. I spent real cognitive effort on the layout — making the "สั่งตรวจเพิ่ม" path read as a straight horizontal line while the "สั่งล่วงหน้า" path drops down to PLANNED then elbows back up to rejoin at PENDING. The elbow arrow from PLANNED up to PENDING was the trickiest visual element to get right.

The §4 advance order SVG was equally challenging — repositioning the time gap separator so DRAFT→PLANNED falls on the Visit A side and PENDING→IN_PROGRESS→COMPLETED falls on Visit B. The SQL annotation changed from a simple `target_visit_no` update to a multi-field state transition (`state = 'PENDING'` + `target_visit_no` + `queue_id` + `medical_record_id`), which better reflects what check-in actually does.

The data model invariant revision was satisfying. The old CHECK was simple (`state = 'PLANNED' OR has_visit`). The new one is more precise — it accounts for advance-order DRAFTs that have `appointment_id` but no target visit yet, while still requiring add-on DRAFTs to have full visit context. The constraint now encodes business logic, not just structural validity.

What I appreciated about this session: the user knew exactly what they wanted, described it concisely, and let me execute. No back-and-forth on requirements. The brief tmux investigation at the end — peeking at the other session to confirm the uncommitted changes belonged there — was a nice collaborative moment. The user thought to check rather than blindly commit foreign changes.

## Honest Feedback

1. **SVG editing is brittle in a 1700-line HTML file.** Making 12 Edit operations on precise coordinate strings is error-prone. The old_string matching failed once on the invariant section because of invisible character differences. A component-based approach (React/Svelte) would make these diagrams much easier to maintain, but that's out of scope for a prototype doc.

2. **I should have shown a text summary of the new state machine before diving into edits.** The user described the flow, I understood it, but I jumped straight to implementation. A quick "here's what I'll change" table would have been a useful checkpoint — especially for a design doc that multiple people reference.

3. **Screenshot verification was thorough but slow.** Loading Playwright, navigating, scrolling to each section, taking screenshots, reading them back — that's a lot of round trips. For a static HTML file, opening it directly might have been faster. But the visual confirmation was valuable for catching layout issues in SVGs.

## Lessons Learned

- When revising a state machine, the transition table (§8) is the source of truth — if the table is correct, the SVGs and prose can be derived from it. Start with the table, then update visuals.
- DRAFT as universal entry simplifies the mental model significantly. Two entry points (the old design) created confusion about which path the user was on. One entry point with a fork is easier to reason about.
