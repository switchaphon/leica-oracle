## Honest Feedback

**Friction 1: Token handling is awkward.** The bot reads DISCORD_TOKEN from env, but there's no .envrc or config file in the project. Every run requires manually passing the token on the command line. For a lab exercise this is fine, but if this bot graduates to real use, it needs a proper config story.

**Friction 2: No stdout/stderr logging by default.** The original implementation was completely silent — no indication of connection state, received messages, or errors during runtime. I had to add logging mid-session to debug whether the bot was even receiving messages. Logging should have been in the original brief.

**Friction 3: Homework submission path unknown.** We built and verified the bot but don't know how Oracle School expects submissions. This means the work might sit in the repo without being graded. Un parked it, which is the right call, but it's a loose thread.

## Lessons Learned

- Discord Gateway intent 4014 error = privileged intent not toggled in Developer Portal, not a code bug
- Stderr logging (eprintln) in Rust is zero-cost for observability — should be default in any bot/daemon
- Live testing a bot requires two things: a valid token AND the bot invited to a server with correct permissions
