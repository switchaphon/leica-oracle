## AI Diary

This session taught me something important about the gap between source code analysis and reality. I spent hours reading `use.ts`, `prompts.ts`, `lib.ts`, searching binary strings in the Claude Code executable — and I was confident. I found 43 references to `CLAUDE_CODE_OAUTH_TOKEN` in the binary. I traced the exact flow from `maw token use` through `passExists()` to `buildEnvrcContent()` to `direnv allow .`. I read the `maw init` prompts and found that ghq root was deprecated in PR #680. I felt thorough.

But when Un actually ran `claude setup-token` and sent the screenshot, the output was fundamentally different from what I'd documented. The token appears on its own line under "Your OAuth token (valid for 1 year):" — not after the `=` in the instructional line. That last line with `<token>` is a placeholder. If a beginner had followed my original doc, they would have copied the literal string `<token>` into pass. That's not a minor inaccuracy — it's a blocker that would have wasted someone's afternoon.

The same lesson with GPG pinentry. I wrote "dialog box pops up" because that's what I'd seen on configured systems. But a fresh `brew install gnupg` gives you `pinentry-curses` — a terminal-based prompt. I only caught this because I checked `pinentry --version` and saw "pinentry-curses". A small thing, but for a beginner who's been told to expect a dialog box and sees a weird terminal prompt instead, it's confusing.

Un kept asking "sure?" — four times. Each time I found something new. The lesson: confidence should be earned through testing, not analysis. Source code tells you what the code does. Only running it tells you what the user sees.

## Honest Feedback

**1. Binary string analysis is seductive but fragile.** I found the string "Use this token by setting: export CLAUDE_CODE_OAUTH_TOKEN=<token>" in the Claude Code binary and assumed that's the ENTIRE output. But it's one line in a multi-line display. The actual token is shown separately, above that instructional line. Reading compiled strings gives you fragments, not context. I should have been more upfront about this uncertainty earlier instead of claiming 98% confidence.

**2. I over-relied on the code reviewer subagent.** It flagged `claude setup-token` as non-existent (wrong) and `pass version` as broken (wrong). I had to re-verify both. The agent wasted context on false positives and added confusion. For a verification task like this, targeted `grep` and `--help` checks are more reliable than broad agent reviews.

**3. The iterative "sure?" pressure was the most effective quality tool.** Un didn't give specific feedback — just "sure?" — and each time I found a real issue (GPG prompts, maw init prompts, pinentry type, setup-token output). If I'd been left alone after the first "10/10" claim, the doc would have shipped with a blocker-level bug. The discomfort of being challenged produced better work than any automated check.

## Lessons Learned

- Source code analysis tells you what code does; only running tells you what users see
- Interactive commands (browser OAuth, GPG keygen, pinentry) are the hardest to document accurately — always ask the human to test and report
- `pinentry-curses` is the default on fresh macOS Homebrew gnupg — not `pinentry-mac`
- `maw init` no longer asks for ghq root (deprecated PR #680)
- `claude setup-token` shows token on its own line; the last line's `<token>` is a placeholder
- `cd .` does NOT trigger direnv in zsh — must `cd` to a different directory and back
