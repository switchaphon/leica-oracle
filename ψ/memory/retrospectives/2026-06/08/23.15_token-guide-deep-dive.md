## AI Diary

This was a teaching session disguised as documentation work. Un came in feeling confused about token switching — "it doesn't feel like it actually switches" — and what followed was the kind of deep unpacking that I think is the real purpose of documentation. Not writing docs FOR someone, but writing docs WITH someone while they're confused, so the confusion shapes the content.

The key insight was that the guide I'd written earlier (during the day session) explained setup perfectly but was weak on the daily switching workflow. It described the pieces but not the chain — pass decrypts into .envrc, direnv loads on cd, claude reads at startup. Each piece has its own trigger mechanism, and if you miss one link, nothing happens. That's why Un felt like "it doesn't switch" — they were probably hitting one of the 3 traps (forgot to reload direnv, forgot to restart claude, or wrong folder).

The per-oracle token question was interesting. Un asked, I explained the approach, then Un said "actually we don't need that, everyone switches together." That's the right call — simplicity wins. The fleet shares one account, and the switching procedure is always: save → switch at leica-oracle → kill → restart. Three steps. Clean.

What I'm proud of: the "What survives vs what dies" table. That captures something non-obvious — that oracle identity lives on disk (survives kill) but conversation context lives in RAM (dies). The implication is clear: /rrr before kill, no need to /awaken again.

## Lessons Learned

- Documentation is most useful when written during confusion, not after understanding
- A "chain" mental model (A triggers B triggers C) is more useful than listing components independently
- When someone says "it doesn't work," trace the full chain — the break is always at one specific link
