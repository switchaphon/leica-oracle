## AI Diary

This was one of those sessions where the real work wasn't writing code — it was making a decision about information architecture. Three files, all claiming to be "the design system," all slowly drifting from each other. DESIGN_SYSTEM.md hadn't been touched since May 5th. DESIGN.md was a snapshot from June 7th. The live DS_MARKDOWN was the only one that actually moved with the code.

The gap audit was sobering. 14 conflicts. Warning color hex didn't match. Card radius values contradicted each other. Queue status colors were described in two completely different visual paradigms (filled hex vs 2-tone outline) across the files. A developer reading DESIGN_SYSTEM.md would build something visually different from what the live DS page shows. That's not a documentation gap — that's a design system actively misleading its consumers.

The grill-me was short — Un immediately agreed to consolidate. "Can we maintain only single as source of truth?" — a question that contains its own answer. The merge itself was straightforward but required judgment: what from the 2065-line philosophy doc actually earns its place in the live export? I kept the soul (tone keywords, fun injection matrix, clinical safety rules, accessibility contrast ratios) and dropped the implementation details (tailwind config patch, responsive breakpoint tables) that belong in code, not in a design doc.

The HTML export was a satisfying technical detail. A simple regex-based markdown-to-HTML converter, wrapped in a styled standalone document with POPS brand gradient and IBM Plex fonts. The iframe preview modal means you can see the rendered HTML without downloading. Small touch, but it makes the "View .html" button actually useful rather than just dumping raw markup.

Net result: -2994 lines of stale static files, +616 lines of living content that can never drift from the page it describes.

## Honest Feedback

**Friction 1: Template literal as document store.** The DS_MARKDOWN lives as a template literal inside a 7700+ line component file. Editing it requires careful escaping awareness (the backtick issue that broke the build on first try). A dedicated .md file imported at build time would be cleaner, but Next.js `'use client'` components can't easily import raw text files without loader config. The tradeoff is real — we chose "always in sync" over "easy to edit."

**Friction 2: Vercel hook false positives.** Every single edit to this file triggered a "RECOMMENDED: External font loader detected" warning about the `@import url(...)` inside the HTML export string. This is a standalone HTML template for offline use — next/font is irrelevant. The hook fired ~8 times this session, each time requiring me to mentally dismiss it. Wasted attention.

**Friction 3: The gap audit should have happened earlier.** Three sources of truth drifting for a month (May 5 to June 8) means any agent or session that read DESIGN_SYSTEM.md during that period got stale guidance. The "post-commit checklist" memory says to re-export DESIGN.md, but nobody was checking DESIGN_SYSTEM.md. The consolidation fixes this going forward, but the drift window was too long.

## Lessons Learned

- **Single source of truth for design tokens must live where it's consumed.** A static .md file that requires manual re-export will always drift. The live page's inline constant can never be out of sync with itself.
- **Gap audits between parallel docs should be periodic, not reactive.** We found 14 conflicts that accumulated silently over 5 weeks.
