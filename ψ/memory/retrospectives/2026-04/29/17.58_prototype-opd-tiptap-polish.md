## Honest Feedback

1. **Block handle padding creates inherent alignment tension.** The NotionEditor's `pl-10` exists to house the `[+][⠿]` drag handle, but it means every parent component must either accept misaligned text or add matching padding. There's no `editorPadding` prop to control this — it's hardcoded. This will bite every new page that uses NotionEditor. A `compact` or `handlePosition` prop would help.

2. **Dev server port conflicts are a recurring time sink.** Having port 3000 occupied by a stale process forces Next.js to port 3002, which breaks NextAuth's NEXTAUTH_URL. The prototype layout's mock session doesn't fully prevent redirects because the root layout's `SessionProviderWrapper` runs independently. This needs either a `NEXTAUTH_URL` env that matches the current port, or the prototype should not inherit the root layout's SessionProvider at all.

3. **Figma MCP output is verbose but loses design intent.** The `get_design_context` tool returned ~300 lines of absolute-positioned divs with Figma node IDs — useful for pixel-perfect reproduction but hard to extract the *design system* from. I had to visually read the screenshot and manually extract the pattern (dark pill = h-6, bg-zinc-900, text-[10px], rounded-md). A "design tokens only" mode would be more practical for iterative work.

## Lessons Learned

- When the editor has built-in padding for UI affordances (block handles), match surrounding content to the editor's indent rather than removing the editor's padding — the affordances exist for a reason.
- Playwright screenshots can lag behind page state due to client-side redirects; using `page.goto()` + `waitForTimeout()` + inline `page.screenshot()` in `browser_run_code` avoids the timing gap.
