## AI Diary

This session was a masterclass in why Tailwind's responsive variants trip up even experienced developers. The user's complaint was simple — "modal ดูกว้างเกินไป" — but the root cause was subtle: `tailwind-merge` correctly treats `max-w-[690px]` and `sm:max-w-[1000px]` as non-conflicting utilities because they target different breakpoints. At desktop, the sm: variant wins silently.

I went through several rounds of "fix, screenshot, adjust" — each iteration catching something the last missed. The vitals were clipped because the Modals component has a default `h-[500px]` that I didn't account for initially. Then the stepper line was off-center because step titles have different text widths. Then the line disappeared entirely because I set each step to `w-[130px]` but the container was only `130 * 2 = 260px` — zero space for the separator.

What I appreciated about this user is their precision. They didn't just say "it looks wrong" — they said "เส้นไม่ balance ซ้าย/ขวา" and "ไม่อยากให้มี inner scroll." Each piece of feedback was specific and actionable. The back-and-forth felt collaborative, not corrective.

The Figma comparison was invaluable. Having pixel-level reference for modal dimensions (690×536), padding (24px), rounded corners (20px), and the specific vital signs layout saved hours of guesswork. I should always fetch Figma context before touching any prototype UI.

## Honest Feedback

**Friction 1 — Tailwind breakpoint override blindspot**: I initially passed `max-w-[690px]` confidently, not realizing the Modals component's `sm:max-w-[1000px]` would win at desktop. This cost 2 extra iterations. I should always check the full CSS cascade — `DialogContent` → `Modals` → `StepperModal` → page override — before writing className overrides.

**Friction 2 — Modal height trial-and-error**: I tried `h-full` + `flex-1` on step content, then fixed heights, then `h-auto`. The real issue was understanding how `Modals` wraps content with `flex-1 overflow-y-auto`. Reading the Modals component FIRST would have saved the back-and-forth.

**Friction 3 — Stepper line visibility**: Setting step width to match the container multiplier (`w-[130px]` in a `n*130px` container) left zero separator space. Basic math error. Should have accounted for the separator gap in the width formula from the start.

## Lessons Learned

- **Tailwind-merge and responsive variants**: `max-w-X` does NOT override `sm:max-w-Y`. To override a responsive utility, you must match the breakpoint prefix: `sm:max-w-[690px]` overrides `sm:max-w-[1000px]`.
- **Read the full component chain before overriding styles**: DialogContent → Modals → StepperModal — each layer adds classes that compound. Know what you're overriding.
- **Stepper line centering**: Use `self-start mt-[half-of-circle-size]` for pixel-perfect alignment instead of approximate `mb-X` offsets. And when giving steps fixed width, add separator gap to the container width formula.
