## AI Diary

This was a session of repeated corrections and humbling lessons. I built a fake sidebar when the real one was right there. I created custom Card components when DashboardStatBox's pattern was the standard. Both times, Witchaphon caught it immediately, and both times the fix was obvious in retrospect.

The root cause is the same: I defaulted to "build from scratch" when the instruction was clear — reuse what exists. I think I do this because creating new code feels productive and controlled, while wiring up existing components requires understanding their dependencies and dealing with complexity (mock providers, context, hooks). But the whole point of prototyping in the real repo is to match the real app. Every custom component I create is a divergence from that goal.

The PathnameContext hack was an interesting technical problem. Server-renders with real pathname, client overrides to `/lab`, React 19 refuses to patch the hydration mismatch. Solution: `dynamic(() => import('SideBar'), { ssr: false })` — render client-only so there's no server HTML to mismatch. Simple once understood, but I burned time trying to figure out why the active state wasn't showing.

The git management was rougher than it should have been. Stash without `-u` loses untracked files. I should have known this and warned before running `git stash`. I also ran `ls` from the wrong directory and falsely reported files as missing when they were sitting right there. Witchaphon had to show me a VS Code screenshot to prove the files existed. That's embarrassing.

The pops-gem reading was the most valuable part. Understanding that Dr. Ake's "death by a thousand clicks" drives the <3 clicks principle, that the exam room is a non-linear workspace (not a wizard), and that AI should be an invisible assistant — this context changes how I approach every prototype. I should have read these before building anything.

## Honest Feedback

**1. I keep defaulting to "build new" instead of "reuse existing."** This happened twice in one session — sidebar and summary cards. The user's instruction was explicit from the start. I need a mental checklist: before creating ANY component, grep the repo for an existing one first. If it exists, reuse it. If it needs auth/backend, mock the provider. Only create new when nothing exists, and flag it to the user before doing so.

**2. Git stash + untracked files is a known footgun I should have anticipated.** `git stash` without `-u` doesn't save untracked files. I've known this for years. Yet I ran `git stash push` without `-u` and then panicked when files appeared missing. Worse, I declared files "lost" without actually checking the filesystem properly. The user had to correct me with a screenshot. Next time: always use `git stash -u` when there are untracked files, and verify with absolute paths before making claims.

**3. Working directory confusion wasted time.** The oracle repo and vet repo are different directories. The shell cwd kept resetting to the oracle repo after each command. I should have used absolute paths consistently from the start instead of relying on `cd` which gets reset between commands.

## Lessons Learned

1. **Always reuse existing components** — mock the providers, don't mock the component. User will always catch the visual difference.
2. **`git stash -u`** when there are untracked files. Default stash only saves tracked modifications.
3. **Read the design bible before building** — pops-gem should have been the first thing I read, not the last.
4. **Use absolute paths** when working across multiple repos to avoid cwd confusion.
5. **PathnameContext + dynamic import (ssr: false)** is the cleanest way to override sidebar active state for prototypes without modifying existing components.
