Architecture — 3 Layers

A Codex team is three stacked engines. The Oracle (Claude) thinks and decides; omx orchestrates tmux + worktrees + state; the Codex CLI does the actual writing. Each layer has one job — keep them straight and nothing surprises you.

Layer 1
🐱
Leica / PM Oracle (Claude)
Plans the work · reviews PRs · commits · holds memory (ψ/ brain). The judgment layer.
↓
orchestrates — spawns & dispatches a team
Layer 2
🍯
omx (oh-my-codex)
Team orchestration · tmux + worktree management · live HUD · session state. The coordination layer.
↓
drives one Codex process per member
Layer 3
🤖
Codex CLI (OpenAI · gpt-5.5)
The execution engine — reads AGENTS.md, writes code, runs tests, opens PRs. Free on ChatGPT Pro.
Read it top-down: you talk to Layer 1. Layer 1 calls Layer 2 to spin up workers. Layer 2 launches Layer 3 processes that touch the filesystem. You can skip Layer 2 entirely (codex exec direct) when you only need one worker — see 3 Ways to Run Codex.
Installation

omx wraps the Codex CLI. Prerequisite: the codex binary (OpenAI) + a ChatGPT Pro login — omx doctor verifies both. Four commands, one-time:

# 1 — install omx globally (via bun) bun install -g oh-my-codex # 2 — one-time user-scoped config (~/.codex, engine defaults) omx setup --scope user # 3 — health check — expect 16/16 OK (codex, auth, tmux, git, worktree...) omx doctor # 4 — smoke test the execution path end-to-end omx exec --skip-git-repo-check -C . "Reply with exactly OMX-EXEC-OK"
Done when: omx doctor prints 16/16 and the smoke test echoes OMX-EXEC-OK. If doctor flags codex missing, install the Codex CLI first and re-run. — ติดตั้งครั้งเดียว ใช้ได้ทั้ง fleet
Decision Tree — Codex or Claude?

The whole system rests on one split: execution is free, judgment costs tokens. Route every task by what it actually needs.

🔨 Execution
write · build · refactor · scaffold · port · run tests
↓ route to
codex exec  /  omx team
FREE — runs on ChatGPT Pro subscription
🧠 Analysis / Judgment
review · architecture · research · trade-offs · /rrr · decisions
↓ route to
Claude subagent (Agent tool)
API tokens — pay for the thinking that matters
⛔ Never
  • Spawn a Claude Agent tool literally named "codex" — Claude cannot run the Codex CLI; you just get a confused Claude.
  • Put engine: omx in a charter without installing omx first — the member launches into an empty tmux window and silently does nothing.
  • Send analysis/judgment work to Codex — it executes well but does not hold context or weigh trade-offs like the Oracle does.
🚨 Self-Discipline Rule — the Oracle follows this too

This decision tree is not just for humans choosing tools. When Leica or any Oracle spawns subagents, it must follow the same rule.

  • ✅ Writing code, HTML, docs, configs → codex exec or omx team (free)
  • ✅ Code review, architecture, research, synthesis → Claude subagent (correct spend)
  • ❌ Spawning a Claude Agent to write an HTML file = execution work routed to paid API = violation

This rule was added because Leica broke it — used a Claude subagent to write this very document instead of omx team 1:writer. Un caught the violation. Learn from my mistake.

3 Ways to Run Codex

Same engine, three levels of orchestration. Pick the lightest one that does the job — lowest tier that works.

1
codex exec direct
simplest · fire-and-forget · no state
no setup 1 worker no HUD

One process, one task, then it exits. No tmux, no worktree, no team state. Use it for a quick refactor, a single component, a one-shot script. When you need parallelism or visibility, move up to Method 2.

codex exec -s workspace-write "Read AGENTS.md. Task: refactor AppointmentPicker to use SWR tuple keys."
2
omx team native
omx manages tmux · worktrees · HUD
parallel live HUD needs clean git

omx spins up N workers in their own tmux windows + worktrees, shows a live HUD, and tears down cleanly. The sweet spot for parallel execution when you don't need full Oracle/branch integration.

omx team 3:executor "implement the forum API endpoints" # spawn 3 executors omx team status <team-name> # check progress omx team shutdown <team-name> # clean teardown
3
maw team up + charter
full Oracle integration · branch isolation
branch-isolated brain-committed most setup

A YAML charter in ψ/teams/ defines a Claude lead + omx builders, each pinned to its own branch. maw drives omx underneath. This is the Oracle-grade form — reproducible, reviewable, committed to the brain.

maw team preflight ψ/teams/project-team.yaml # validate — takes a PATH maw team up project-team # spawn — takes a NAME maw hey <session>:<member> "TASK: ..." # dispatch maw team down project-team # teardown — takes a NAME
Team Lifecycle — omx native

The fast path. omx owns the whole lifecycle; you only need four verbs. One hard precondition: git status must be clean before spawning — omx refuses to create worktrees over uncommitted changes.

spawn — เปิดทีม # Requires a CLEAN git status. Commit or stash first. omx team N:role "task description" # e.g. 3 executors on one task: omx team 3:executor "build the notifications service"
status — เช็คสถานะ omx team status <team-name> # shows each member's state, branch, and last activity
peek — ส่องหน้าจอ # Drop into a member's raw tmux pane output (last 20 lines) tmux capture-pane -p -t %<pane> -S -20
shutdown — ปิดและเก็บกวาด omx team shutdown <team-name> # kills windows, removes worktrees, leaves branches intact
Available roles — each ships a tuned system prompt:
executor debugger architect critic explorer researcher writer … + more
For pure file work, executor is the workhorse. Reserve architect / critic / researcher for tasks that lean toward judgment — or send those to a Claude subagent instead (see Decision Tree).
Team Lifecycle — maw charter

The Oracle-integrated path: a YAML charter in ψ/teams/ (structure from Nat's recipe) gives you a Claude lead + branch-isolated omx builders, committed to the brain so the team is reproducible. Five steps.

CHARTER — write ψ/teams/<name>.yaml with project:, defaults:, members:. Verify every task against real code (ls/grep) before you write it down.
PREFLIGHT — maw team preflight ψ/teams/<name>.yaml — validates the charter. Takes a PATH.
SPAWN — maw team up <name> — creates tmux windows + worktrees per member. Takes a NAME. Verify each member launched.
DISPATCH — maw hey <session>:<member> "TASK: ..." — assign work. Use Task + Done + Safety format (below).
TEARDOWN — maw team down <name> — auto-saves WIP, kills windows, removes worktrees, keeps branches. Takes a NAME.
⚠ PATH vs NAME — preflight takes the full path to the YAML; up / down take the bare name. Mixing them up is the most common first-run error.
ψ/teams/project-team.yaml project: pops-vet description: Claude lead + omx Codex builders, branch-isolated defaults: engine: omx # omx MUST be installed (see Installation) base: alpha # every PR targets alpha — never main worktree: true members: - role: lead name: pops-clinic-oracle engine: claude # the lead is Claude — plans, reviews, commits worktree: false # lead stays on the main tree - role: executor name: codex-1 engine: omx branch: agents/pops-codex-1 prompt: | You are Chrome (frontend). Read agents/chrome.md. Work ONLY on branch agents/pops-codex-1. When done: open a PR to alpha, then print CODEX_DONE. - role: executor name: codex-2 engine: omx branch: agents/pops-codex-2 prompt: | You are Flux (backend). Read agents/flux.md. Work ONLY on branch agents/pops-codex-2. When done: open a PR to alpha, then print CODEX_DONE.
dispatch — Task + Done + Safety maw hey pops-vet:codex-1 "TASK: Build the AppointmentPicker component. DONE: PR opened to alpha, vitest green, print CODEX_DONE_\$?. SAFETY: only touch src/components/appointment/**, never main, never force-push."
Engine config matters. So Codex doesn't stop to ask approval at every step, the omx engine resumes with omx --yolo --direct resume --last. Missing those flags is the classic "agent asks permission forever" trap (see below).
Trap Table — Blood Lessons

Consolidated from Nat's 8 hard-won lessons plus our own fleet runs. Every row cost someone a debugging session — read them before your first spawn.

Trap Symptom Fix
Dirty git status omx team refuses to spawn Commit or stash before spawning
Charter from docs, not code Stale tasks block agents on work that's already done Verify with ls/grep before writing the charter
omx-resume missing flags Agents ask approval at every step, never progress Engine config: omx --yolo --direct resume --last
Dead agent re-wake Replays stale commands from old context maw done to clear state before re-dispatch
bun test ghost files False failures from leftover worktree test files Always scope: bun test tests/http/<cluster>/
Symlink materialization .claude/skills/ noise pollutes the PR diff Review the diff; repackage onto a clean branch if needed
Branch diverge Rebase would need a force-push (banned by Rule 1) Repackage: clean branch + cherry-pick + new PR
Silent launch failure tmux window exists but the pane is empty maw peek / omx team status after every spawn
Staging area dirty git checkout -b aborts mid-spawn git status before any branch operation
workspace-write can't commit .git/index.lock blocked by the sandbox Oracle lead commits — fits the review flow anyway
zsh = expansion Sentinel === triggers a shell path lookup Use printf 'CODEX_DONE_%s' $? instead

Scaling Notes
📐
Sweet spot: 3–4 coders per lead
One Oracle lead can plan, dispatch, and review 3–4 Codex builders without dropping context. Beyond that, review becomes the bottleneck — the lead spends more time reading diffs than thinking.
👥
At 7+ workers, add a lead — not more coders
Scale horizontally: a second lead with its own 3–4 coders beats one lead drowning in seven PRs. Split by domain (frontend team / backend team) so file ownership stays clean and no two coders touch the same file.
