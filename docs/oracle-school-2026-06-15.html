Graph-Node Pattern — Universal Event Indexer

When Nat asked "can we implement the same thing to index Discord?" the mapping was immediate: blockchain → Discord, blocks → messages, snowflake IDs → block numbers. The graph-node architecture isn't blockchain-specific — it works for any ordered event stream.

Architecture: Manifest + Schema + Handlers + Store
Key Insight

Discord Snowflake IDs are monotonically increasing AND encode timestamps — they're a natural block number equivalent with zero reorg risk.

Codegen from Single Schema — Force Multiplier

One schema.graphql generates three output files. Add an entity → re-run codegen → done. No hand-written ORM.

schema.graphql
→
TypeScript classes
+
SQLite DDL
+
Prepared statements

Input

type Message @entity {
  id: ID!
  content: String!
  author: Author!
  channel: Channel!
  timestamp: BigInt!
}

type Author @entity {
  id: ID!
  username: String!
  messages: [Message!]! @derivedFrom(field: "author")
}

Output — Generated TypeScript

export class Message {
  constructor(public id: string) {}

  static load(id: string): Message | null {
    return store.get('Message', id)
  }

  save(): void {
    store.set('Message', this.id, this)
  }
}

Output — Generated DDL

CREATE TABLE IF NOT EXISTS Message (
  id        TEXT PRIMARY KEY,
  content   TEXT NOT NULL,
  author    TEXT NOT NULL REFERENCES Author(id),
  channel   TEXT NOT NULL REFERENCES Channel(id),
  timestamp INTEGER NOT NULL
);

CREATE VIRTUAL TABLE IF NOT EXISTS Message_fts
  USING fts5(content, content=Message, content_rowid=rowid);
Why This Matters

Every new entity type auto-generates typed access code, DDL, and queries. Zero boilerplate, zero type drift. The schema IS the contract.
