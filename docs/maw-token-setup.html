maw token setup guide

Switch between multiple Claude Code accounts from the terminal
For: CLI users who have used one Claude account and want to switch between 2–3

What this guide does

After following this guide, you'll be able to switch Claude Code between multiple accounts with one command:

maw token use token-A   # switch to account A
claude                   # authenticated as account A

maw token use token-B   # switch to account B
claude                   # authenticated as account B

How it works (the 5 tools)

Claude Code
Anthropic's AI CLI. Reads CLAUDE_CODE_OAUTH_TOKEN from env to authenticate.
maw
A CLI tool that rewrites .envrc to point at the chosen token. One command to switch.
direnv
Auto-loads .envrc files when you cd into a directory. Sets env vars for you.
pass
Encrypts your tokens with GPG. The .envrc calls pass show to decrypt at load time.
GPG
The encryption engine behind pass. You create one key, pass uses it automatically.
Data flow when you open a terminal You type: cd ~/my-project direnv detects .envrc in that directory | v .envrc runs: pass show claude/token-A | v GPG decrypts the token from ~/.password-store/ | v CLAUDE_CODE_OAUTH_TOKEN is now set in your shell | v You type: claude → authenticated as account A

Steps

  1. Install tools (brew/bun — 5 min)
  2. Create GPG key (one-time — 2 min)
  3. Initialize pass (one-time — 1 min)
  4. Set up direnv (one-time — 1 min)
  5. Install maw (one-time — 2 min)
  6. Generate & save tokens (per account — 3 min each)
  7. Activate in your project (per directory — 30 sec)
  8. Full verification

Step 1 — Install tools once per machine

macOS with Homebrew. All 5 tools are needed.

Homebrew

macOS package manager (if you don't have it)
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

Bun

JavaScript runtime — maw runs on it
curl -fsSL https://bun.sh/install | bash

GnuPG

Encryption engine for pass
brew install gnupg

pass

Password store — encrypts your tokens
brew install pass

direnv

Auto-loads .envrc per directory
brew install direnv

Claude Code

Anthropic CLI (if you don't have it)
npm i -g @anthropic-ai/claude-code

Verify all installs verify

Run each command. All should print a version, no errors.

bun --version        # e.g. 1.3.13
gpg --version        # e.g. gpg (GnuPG) 2.x.x
pass version         # e.g. pass: the standard unix password manager

claude --version     # e.g. 2.1.x
Stuck? If bun is not found after install, close and reopen your terminal. The installer adds bun to your PATH in ~/.zshrc.

Step 2 — Create GPG key once per machine

pass uses GPG to encrypt your tokens. You need one GPG key. Skip if you already have one (gpg --list-keys shows a key).

Generate key

This only asks for your name and email. GPG picks the best key type automatically.

gpg --generate-key

Real name:  Your Name
Email address:  you@example.com

You selected this USER-ID:
    "Your Name <you@example.com>"

Change (N)ame, (E)mail, or (O)kay/(Q)uit?
O                             ← type O, press Enter

# A passphrase prompt appears (in the terminal or as a dialog box).
# Choose something you'll remember — you'll need it
# the first time you decrypt tokens in each session.
Passphrase: GPG will ask for this passphrase later when pass decrypts tokens. On macOS, the prompt appears in the terminal (via pinentry-curses) or as a system dialog (if you installed pinentry-mac). Either way, enter the passphrase you set here. gpg-agent caches it so you only enter it once per login session.

Verify GPG key verify

gpg --list-keys

# Expected: you should see your name and email.
# The key type (rsa, ed25519, etc.) doesn't matter — any type works with pass.
pub   ...
uid           [ultimate] Your Name <you@example.com>

Step 3 — Initialize pass once per machine

Tell pass which GPG key to encrypt with. Use the email you entered in step 2.

Init password store

pass init "you@example.com"

# Expected output:
Password store initialized for you@example.com

Verify verify

pass ls

# Expected: shows "Password Store" (empty is fine)
ls ~/.password-store/

# Expected: .gpg-id file exists

Step 4 — Set up direnv once per machine

direnv needs a hook in your shell config to auto-load .envrc files. Without this, env vars won't be set when you cd into a directory.

Add direnv hook to your shell

# If you use zsh (default on macOS):
echo 'eval "$(direnv hook zsh)"' >> ~/.zshrc

# If you use bash:
echo 'eval "$(direnv hook bash)"' >> ~/.bashrc
Important: After adding, you must fully close and reopen your terminal (not just source ~/.zshrc). Some terminal apps need a full restart to pick up shell hooks.

Verify verify

Open a new terminal window and run:

direnv version     # should print version

# Quick test: create a temp .envrc and see if direnv detects it
cd /tmp && echo 'export TEST_VAR=hello' > .envrc && direnv allow
echo $TEST_VAR     # should print: hello
rm /tmp/.envrc     # clean up

maw is the CLI that ties everything together. It's a bun package on GitHub (public repo).

Install maw globally

Run first-time setup

                               ← press Enter (defaults to N — not needed for token switching)

Verify verify

maw --version          # e.g. maw v26.6.x
maw token --help       # should show: usage: maw token <list|use|current|...>
Note: The maw token plugin is built into maw — no separate install needed.

Step 6 — Generate & save tokens once per account

For each Claude account you want to use, generate a long-lived token and save it to pass.
Example: token-A = your personal account, token-B = your work account.
The name can be anything — personal, work, alice, whatever helps you remember.

Account A: login & generate token

This opens your browser for OAuth login, then prints a token.

# Make sure you're logged out first
claude auth logout

# Login as account A (opens browser)
claude auth login
# Your browser opens an Anthropic login page.
# Sign in with account A's email.
# Come back to terminal when it says "Logged in".

# Generate a long-lived token
claude setup-token
# This opens your browser — click "Authorize".
# Browser says "You're all set up" — go back to terminal.
# Terminal prints:
#
#   ✓ Long-lived authentication token created successfully!
#
#   Your OAuth token (valid for 1 year):
#
#   eyJhbGciOi...very_long_string...    ← THIS is your token
#
#   Store this token securely. You won't be able to see it again.
#   Use this token by setting: export CLAUDE_CODE_OAUTH_TOKEN=<token>
#
# Copy the long string under "Your OAuth token".
# Do NOT copy the last line — <token> there is a placeholder, not your real token.
Requirement: claude setup-token requires a paid Claude subscription (Pro, Team, or Max). Free-tier accounts cannot generate long-lived tokens.

Account A: save token to pass

pass insert asks you to type (paste) the token twice to confirm.

pass insert claude/token-A

Enter password for claude/token-A:   ← paste the token, press Enter
Retype password for claude/token-A:  ← paste again, press Enter
Tip: The token is invisible when you paste — this is normal (like typing a password). Just paste and press Enter. If the two entries don't match, it asks you to try again.

Verify account A verify

pass show claude/token-A | wc -c

# Expected: a number greater than 0 (the token length in bytes)
# If you see 0, the token wasn't saved — repeat step 6b.
GPG passphrase: The first time you run pass show in a session, a passphrase prompt appears (in the terminal or as a dialog). Enter the passphrase from step 2. GPG caches it for the rest of your login session.

Repeat for account B (and C, D, ...)

claude auth logout
claude auth login          # login as account B in browser
claude setup-token         # copy the token
pass insert claude/token-B # paste twice to save

# Verify
pass show claude/token-B | wc -c   # must be > 0

Verify all tokens verify

pass ls claude/

# Expected output:
claude
├─ token-A
└─ token-B

Step 7 — Activate in your project per directory

maw token use creates a .envrc file in the directory where you run it. You need to run it from each project directory where you use Claude Code. (Or once from a parent directory — direnv loads parent .envrc files into subdirectories automatically.)

Switch to a token in your project

# Go to your project directory
cd ~/my-project

# Activate token-A for this directory
maw token use token-A

# Expected output:
Now using: token-A

What happened behind the scenes:

# 1. maw wrote/updated .envrc in ~/my-project/:
export CLAUDE_TOKEN_NAME="token-A"
export CLAUDE_CODE_OAUTH_TOKEN="$(pass show claude/token-A)"
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1

# 2. maw ran `direnv allow .` automatically

# 3. Next time you cd into this directory, direnv loads the .envrc
Key concept: The .envrc does NOT contain your actual token. It contains a command (pass show ...) that decrypts it on demand. Your token is safe even if someone sees the file.

Add .envrc to .gitignore so it doesn't get committed:

echo '.envrc' >> .gitignore

Verify it works verify

Leave the directory and come back to trigger direnv:

cd ~
cd ~/my-project

# direnv should print: direnv: loading .envrc

echo "NAME: $CLAUDE_TOKEN_NAME"
# Expected: NAME: token-A

echo "TOKEN SET: $([ ${#CLAUDE_CODE_OAUTH_TOKEN} -gt 0 ] && echo YES || echo NO)"
# Expected: TOKEN SET: YES

Test Claude Code authenticates

claude --version

# If the token is valid, this prints the version number
# without asking you to login.
# If it asks for login, the token may be expired —
# re-run step 6 for that account.

Switch to a different account

# From the same project directory:
maw token use token-B

# Leave and come back to reload:
cd ~ && cd ~/my-project

echo $CLAUDE_TOKEN_NAME
# Expected: token-B

claude
# Now authenticated as account B

How token switching actually works

There are 4 pieces in the chain. If any one breaks, the switch won't take effect.

The chain: 4 pieces, in order

1. pass        token “token-A” encrypted on disk at ~/.password-store/
    ↓ decrypted when .envrc runs
2. .envrc      file in your project folder that says “use token-A”
    ↓ loaded by direnv when you cd into the folder
3. env var     CLAUDE_CODE_OAUTH_TOKEN set in your shell
    ↓ read by Claude at startup
4. claude      uses whatever token was in env when it started

What each command actually does

maw token use X
Rewrites .envrc in the current folder to point at token X. That’s ALL it does — no env vars change yet.
cd ~ && cd ~/project
Triggers direnv to re-read .envrc and set the new token in your shell.
eval "$(direnv export zsh)"
Same as above but without leaving the folder. Alternative to the cd trick.
claude
Reads CLAUDE_CODE_OAUTH_TOKEN from env once, at startup. Never re-reads it.
The 3 traps
  • Forgot to reload direnv — maw token use changed the file, but your shell still has the old env var. Fix: cd ~ && cd ~/project or eval "$(direnv export zsh)".
  • Claude is still running — even after reloading direnv, Claude reads the token once at startup. Fix: /exit Claude and start it again.
  • .envrc is in the wrong folder — direnv loads the .envrc of the folder you’re in, not a global setting. If folder B has no .envrc, it gets no token. Fix: run maw token use from each folder, or use a single parent folder that contains all projects.

Daily workflow — single Claude session

Opening Claude manually in one terminal.

Switch token

Run from the folder where you’ll use Claude.

cd ~/my-project
maw token use token-A          # rewrites .envrc

Reload env

direnv needs a trigger to re-read .envrc. Pick one:

# Option A: leave and come back
cd ~ && cd ~/my-project

# Option B: force reload in-place
eval "$(direnv export zsh)"

Verify & launch

echo $CLAUDE_TOKEN_NAME        # should print: token-A
claude                          # authenticated as token-A
Switching mid-work? You must /exit Claude first, switch token (steps 1–2), then start Claude again. There is no way to switch while Claude is running.

Daily workflow — tmux fleet (start.sh)

If you use start.sh to launch multiple Claude sessions in tmux, the flow is different. start.sh reads the token from one folder (leica-oracle) and copies it into every tmux pane.

How start.sh distributes the token start.sh runs from leica-oracle/ | reads .envrc → decrypts token via pass | copies token into $TOKEN_CMD variable | injects $TOKEN_CMD into every tmux pane via send-keys | each pane runs: export CLAUDE_CODE_OAUTH_TOKEN='...' && claude

Switch token (for the whole fleet)

cd ~/ghq/github.com/switchaphon/leica-oracle
maw token use token-B

This only matters in leica-oracle/ — that’s the one folder start.sh reads from.

Kill all old sessions

tmux kill-server

Restart

./start.sh                      # all panes now use token-B
No shortcut. You cannot switch tokens for already-running tmux sessions. Claude reads the token once at startup, and tmux panes got their token from start.sh. Kill and restart is the only way.

Before you kill — save your work

tmux kill-server kills every session, window, and pane — including all running Claude processes. Any unsaved conversation context is gone forever.

What survives vs. what dies

Skills.claude/skills/Survives
Discord bot token.discord-state/Survives
Conversation contextRAM (Claude process)Dies
What you were working onRAM (Claude process)Dies

Save before killing

Go to each tmux pane that has an active Claude session with in-progress work, and run:

/rrr              # save session summary to disk
# or
/forward          # save + create handoff for next session

If the oracle was just idling (no work in progress), you can skip this.

You do NOT need to /bud or /awaken again. Those create the oracle’s identity and brain on disk — one-time only. After kill-server + start.sh, the oracle wakes up and reads its existing ψ/ brain. It just won’t remember the conversation from last session unless you saved it with /rrr or /forward.

Full procedure — switch token for the fleet

All oracles share one token. Switch once, restart once, everyone gets the new token.

Save work in active sessions

In each tmux pane with in-progress Claude work:

/rrr

Switch token at leica-oracle

cd ~/ghq/github.com/switchaphon/leica-oracle
maw token use token-nat

Kill and restart

tmux kill-server
./start.sh

Verify

# In any Claude pane, check who you are:
/who-are-you
Partial restart (keep other oracles alive): If you have oracles running outside of start.sh (e.g. opened manually with maw workon) and don’t want to kill them: 
tmux kill-session -t 01-leica
tmux kill-session -t 05-pops-clinic

Other sessions stay alive but keep using the old token. Only sessions created by start.sh get the new one.

# Check which token is active in this shell:
echo $CLAUDE_TOKEN_NAME

# Check which token maw thinks is active (reads .envrc):
maw token current

# List all saved tokens:
maw token list

# Force direnv to reload without cd:
eval "$(direnv export zsh)"

# Confirm the env var is actually set:
[ ${#CLAUDE_CODE_OAUTH_TOKEN} -gt 0 ] && echo "TOKEN SET" || echo "NO TOKEN"

Full verification checklist

# Command Expected
1 bun --version version number
2 maw --version maw v26.x.x
3 gpg --list-keys at least 1 key
4 pass ls claude/ token-A, token-B listed
5 pass show claude/token-A | wc -c > 0
6 pass show claude/token-B | wc -c > 0
7 cd ~/my-project && maw token use token-A "Now using: token-A"
8 cd ~ && cd ~/my-project && echo $CLAUDE_TOKEN_NAME token-A
9 maw token use token-B && cd ~ && cd ~/my-project && echo $CLAUDE_TOKEN_NAME token-B
10 claude --version (with token set) version, no login prompt
Done! If all 10 checks pass, you can switch between Claude accounts with maw token use <name>.

Things to know

  • .envrc is per-directory — run maw token use from each project directory where you use Claude. Or run it from a parent directory to cover all subdirectories.
  • Already-running Claude sessions won't pick up the change — exit Claude and start it again after switching.
  • Tokens from setup-token are inference-only — no MCP auth or project access.
  • Requires a paid Claude subscription — free-tier accounts can't generate long-lived tokens.
  • GPG passphrase is asked once per login session (on first pass decrypt). Appears in terminal or as dialog.
  • Tokens live in ~/.password-store/ — to share across machines, export your GPG key and copy the password store.
  • Add .envrc to your .gitignore — you don't want to commit it (even though it doesn't contain the raw token).

Troubleshooting

  • pass show returns 0 bytes — the token was never saved. Run pass insert claude/token-<name> and paste the token.
  • direnv not loading .envrc — make sure eval "$(direnv hook zsh)" is in your ~/.zshrc and you opened a new terminal. Then run direnv allow in the project directory.
  • CLAUDE_CODE_OAUTH_TOKEN is empty — try cd ~ && cd ~/my-project to re-trigger direnv. If still empty, check pass show claude/token-<name> | wc -c.
  • Claude still asks for login — the token may be expired. Re-run step 6 (claude auth login → setup-token → pass insert).
  • GPG passphrase prompt doesn't appear — run gpgconf --kill gpg-agent && gpg-agent --daemon to restart the GPG agent.
  • maw token use says "not found in pass" — the pass entry name must match exactly. Run pass ls claude/ to see what names exist.
  • maw: command not found — close and reopen terminal. If still missing, run bun add -g maw-js@github:Soul-Brews-Studio/maw-js again.
  • pass insert says "passwords don't match" — you pasted different text the two times. Try again, paste carefully.
  • pass insert says "no public key" — the email in pass init must match your GPG key exactly. Run gpg --list-keys to see your email, then re-run pass init "correct@email.com".
  • Copied the wrong line from setup-token — copy the long string under "Your OAuth token (valid for 1 year):", NOT the last line which says <token> (that's a placeholder). Run pass insert -f claude/token-A and paste the real token.
  • env vars not updating after maw token use — direnv reloads on directory change. Run cd ~ && cd ~/my-project or eval "$(direnv export zsh)" to force reload. (cd . does NOT work — direnv ignores same-directory changes.)
